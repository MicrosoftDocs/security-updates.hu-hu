Security Bulletin

A Microsoft számú biztonsági közleménye MS11-086 - Fontos
=========================================================

Az Active Directory biztonsági rése jogok kiterjesztését teheti lehetővé (2630837)
----------------------------------------------------------------------------------

Közzétéve: 2011. november 8.

**Verzió:** 1.0

### Általános tudnivalók

#### **Összefoglalás**

A biztonsági frissítés az Active Directory, az Alkalmazásmódú Active Directory (ADAM) és az Active Directory Lightweight Directory-szolgáltatás (AD LDS) közvetlenül jelentett biztonsági rését szünteti meg. A biztonsági rés a jogok kiterjesztését teszi lehetővé, ha az Active Directory szolgáltatást úgy konfigurálja, hogy az használja az LDAP SSL protokollon keresztül (LDAPS) protokollt, és a támadó érvényes tartományi fiókhoz társított, visszavont tanúsítványt szerez, majd azzal hitelesíti magát az Active Directory tartományon. Alapértelmezés szerint az Active Directory szolgáltatásban nincs konfigurálva az LDAPS protokoll használata.

Ez a biztonsági frissítés fontos besorolású a Windows XP, Windows Server 2003, Windows Vista, Windows Server 2008 (kivéve Itanium), Windows 7 és Windows Server 2008 R2 (kivéve Itanium) összes támogatott kiadására telepített Active Directory, ADAM és AD LDS szolgáltatásokra nézve. További tudnivalókat a közlemény későbbi részében az **Érintett és nem érintett szoftverek** cím alatt talál.

A biztonsági frissítés a biztonsági rés megszüntetése érdekében módosítja a tanúsítványok visszavont tanúsítványok listáján (CRL) történő ellenőrzési módját az Active Directory szolgáltatásban. A biztonsági réssel kapcsolatban bővebben az adott problémával foglakozó gyakran feltett kérdéseket ismertető szakaszban, **A biztonsági réssel kapcsolatos tudnivalók** című részben olvashat.

**Javaslat:** A felhasználók nagy részénél be van kapcsolva az automatikus frissítés; ilyenkor nincs más teendő, mert a rendszer automatikusan letölti és telepíti a biztonsági frissítést. Ha nem használ automatikus frissítést, keresse meg a vonatkozó frissítést és telepítse manuálisan. Az automatikus frissítés további beállításairól lásd a [Microsoft Tudásbázis 294871. cikkét](http://support.microsoft.com/kb/294871).

Ha a rendszergazda, vállalati telepítő vagy végfelhasználó manuálisan szeretné telepíteni a biztonsági frissítést, a Microsoft a minél előbbi telepítést javasolja frissítéskezelő szoftver vagy a [Microsoft Update](http://go.microsoft.com/fwlink/?linkid=40747) szolgáltatás igénybevételével.

Lásd még a közlemény **Észlelési és telepítési eszközök, útmutatás** fejezetét.

**Ismert problémák** Nincsenek

Az LDAPS hitelesítés kihagyása miatti biztonsági rése - CVE-2011-2014
---------------------------------------------------------------------

<span></span>
Az Active Directory szolgáltatásban jogok kiterjesztését lehetővé tévő biztonsági rés található, ha úgy konfigurálták, hogy használja az LDAP SSL protokollon keresztül (LDAPS) protokollt. A támadó úgy használhatja ki a biztonsági rést, ha korábban visszavont tanúsítvánnyal hitelesíti magát az Active Directory tartományon, így hozzáférést szerez a hálózati erőforrásokhoz, vagy programkódot futtat a tanúsítványhoz társított, meghatározott jogosult felhasználó jogosultságainak felhasználásával.

A biztonsági rés standard bejegyzésének megtekintéséhez a gyakran előforduló biztonsági rések besorolási listáján lásd: [CVE-2011-2014](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2011-2014).

#### Az LDAPS hitelesítés kihagyása miatti biztonsági résének (CVE-2011-2014) súlyosságát enyhítő tényezők

Az enyhítő tényezők közé sorolható egy beállítás, konfiguráció vagy általánosan bevált gyakorlat, amely alapértelmezett állapotában csökkentheti a biztonsági rés kihasználásának valószínűségét. A következő enyhítő tényezők hasznosnak bizonyulhatnak:

-   A tűzfalakra vonatkozó gyakorlati tanácsok, valamint a szabványos alapértelmezett tűzfal-konfigurációk segítséget nyújtanak a hálózatot érő, a szervezet határain kívülről eredő támadások kivédésében. Gyakorlati tanácsként azt javasoljuk, hogy az internetkapcsolattal rendelkező számítógépeken a lehető legkevesebb port legyen megnyitva.

#### Az LDAPS hitelesítés kihagyása miatti biztonsági résének (CVE-2011-2014) lehetséges megoldásai

A lehetséges megoldás olyan beállítást vagy konfigurációs módosítást takar, amely magát a biztonsági rést nem szünteti meg, azonban a frissítés telepítése előtt segít az ismert támadási lehetőségek kivédésében. A Microsoft alaposan megvizsgálta a következő lehetséges megoldásokat, és tájékoztatja a felhasználókat, amennyiben a megoldás csökkenti a működőképességet:

-   **Távolítsa el** **a felhasználói** **fiókot** **a tartományból, amelyre vonatkozott a visszavont** **tanúsítvány.**
-   **Az LDAPS használata mellett Használjon IPsec alapú kapcsolatokat a "CRL ellenőrzése mindig" üzemmódban.**

    Az IPsec segítségével történő CRL ellenőrzés módjával kapcsolatos információkat a [Visszavont tanúsítványok listájának (CRL) ellenőrzése - alkalmazás választása](http://social.technet.microsoft.com/wiki/contents/articles/964.aspx) TechNet cikk tartalmaz.

    **Megjegyzések**

    1.  Az IPsec használatával titkosíthatja és védheti meg a tanúsítványhitelesítést és CRL ellenőrzést használó LDAPS kapcsolatait.
    2.  Az IPsec használatával védheti meg a tanúsítványhitelesítést és CRL ellenőrzést használó, titkosítás nélküli LDAPS kapcsolatait. Ebben az esetben az LDAPS kezeli a titkosítást.

        A Windows Vista, Windows Server 2008, Windows 7, és Windows Server 2008 R2 érintett kiadásaiban az AuthIP (a felhasználóhitelesítést lehetővé tevő IPsec bővítmény) megakadályozhatja, hogy a támadó számítógép-tanúsítvánnyal IPsec tanúsítványhitelesítést, majd a visszavont felhasználói tanúsítvány használatával LDAPS-hitelesítést hajtson végre.

    3.  A kétszeres titkosítás elkerüléséhez az LDAPS esetében használjon NULL-beágyazást, és vegye figyelembe, hogy az IPsec szintjén történő tanúsítványhitelesítés független az SSL-szintű hitelesítésétől. Ez azt jelenti, hogy különböző tanúsítványok használata az egyes szinteken még lehetővé teheti az LDAPS elérését a lejárt tanúsítványon keresztül.

#### Gyakori kérdések az LDAPS hitelesítés kihagyása miatti biztonsági résével (CVE-2011-2014) kapcsolatban

**Mire terjed ki a biztonsági rés hatása?**
A biztonsági rés jogok kiterjesztését teheti lehetővé. A biztonsági rést sikeresen kihasználó támadó hálózati erőforrásokhoz kaphat hozzáférést vagy kódot futtathat egy konkrét jogosult felhasználó jogosultságainak felhasználásával.

**Mi okozza a biztonsági rést?**
A biztonsági rés akkor jelentkezik, amikor az LDAPS protokoll használatára konfigurált Active Directory szolgáltatás nem tudja ellenőrizni az SSL-tanúsítványok visszavont állapotát a tartományi fiókhoz tartozó CRL listához képest. Így a rendszer a tanúsítványokat azután is elfogadhatja érvényesként, miután a hitelesítésszolgáltató (CA) visszavonta őket.

**Mit jelent a hitelesítésszolgáltató (CA)?** **
**A hitelesítés-szolgáltatók azok a szervezetek, akik kiadják a tanúsítványok. Megállapítják és ellenőrzik a személyek vagy hitelesítés-szolgáltatók tulajdonát képező nyilvános kulcsot hitelességét, illetve ellenőrzik a tanúsítványt kérő személy vagy szervezet azonosságát.

**Mik azok** **azActive Directory** **szolgáltatások?**
Az Active Directory szolgáltatások egy bővíthető és méretezhető szolgáltatáskészlet, melynek segítségével hatékonyan kezelhetők a vállalati azonosítók, a hitelesítő adatok, az adatvédelem, valamint a rendszer- és alkalmazásbeállítások. Az Active Directory szolgáltatások közé tartozik többek között az Active Directory Tanúsítványszolgáltatás (AD CS), az Active Directory tartományi szolgáltatások (AD DS), az Active Directory összevonási szolgáltatások (AD FS), az Active Directory Lightweight Directory-szolgáltatások (AD LDS) és az Active Directory tartalomvédelmi szolgáltatások (AD RMS).

**Mik azok az Active Directory Lightweight Directory-szolgáltatások** **(AD LDS)?
**Az Active Directory Lightweight Directory-szolgáltatások (AD LDS) az Active Directory független üzemmódját jelentik, amely célzott címtárszolgáltatásokat nyújt az alkalmazásokhoz. Az AD LDS a Windows Vista, Windows Server 2008, Windows 7 és Windows Server 2008 R2 rendszereken érhető el. Az AD LDS az Alkalmazásmódú Active Directory (ADAM) szolgáltatást váltja fel, amely a Windows XP és a Windows Server 2003 rendszereken használható. Az AD LDS szolgáltatásról bővebben az [Active Directory Lightweight Directory-szolgáltatások](http://technet.microsoft.com/en-us/library/cc731868(ws.10).aspx) című TechNet-cikkben olvashat.

**Mi az Alkalmazásmódú Active Directory?** **
**Az Alkalmazásmódú Active Directory (ADAM) a Windows XP és a Windows Server 2003 rendszereken elsősorban felhasználói szolgáltatásként, másodsorban rendszerszolgáltatásként futó Lightweight Directory Access Protocol (LDAP) címtárszolgáltatás. Az ADAM szolgáltatásról a [Bevezetés az ADAM használatába](http://technet.microsoft.com/en-us/library/cc780837(ws.10).aspx) című TechNet-cikkben talál bővebb információt.

**Mi a Lightweight Directory Access Protocol (LDAP)?
**A Lightweight Directory elérési protokoll (LDAP) egy címtár-szolgáltatási protokoll, amely közvetlenül a TCP/IP verem fölött fut. Az LDAP információs modellje (adatok és névterek esetében egyaránt) hasonló az X.500 OSI címtárszolgáltatáshoz, de kevesebb szolgáltatással és alacsonyabb erőforrás-igénnyel, mint az X.500. A legtöbb más internet-protokolltól eltérően az LDAP-nak van egy társított alkalmazásfejlesztési felülete (API), amely leegyszerűsíti az internetes címtár-szolgáltatási alkalmazások írását. Az LDAP API alkalmazható címtárkezelésre és böngészőalkalmazásokra, amelyeknek nincs címtár-szolgáltatási támogatásuk elsődleges funkcióként. Az LDAP nem képes létrehozni címtárakat, vagy meghatározni, hogyan működjön egy címtárszolgáltatás.

**Mi a visszavont tanúsítványok listája (CRL)?** **
**A CRL egy digitális aláírással ellátott lista, amit a CA ad ki az általa kiadott, és nemrégiben visszavont tanúsítványok felsorolásával. Minden egyes visszavont tanúsítványról közlik a tanúsítvány sorozatszámát, a visszavonás dátumát és okát. Az alkalmazások a CRL alapján tudják megállapítani, hogy egy adott tanúsítványt visszavontak-e. A visszavont tanúsítvány ellenőrzésének részleteiről lásd a [Tanúsítvány-visszavonás és állapot-ellenőrzés](http://technet.microsoft.com/en-us/library/ee619730(ws.10).aspx) című TechNet cikket.

**Mit jelent az LDAP SSL protokollon keresztül** **(LDAPS)?
**Az LDAP SSL protokollon keresztül (LDAPS) protokoll Secure Sockets Layer (SSL)/Transport Layer Security (TLS) technológiával titkosítja az ügyfél- és kiszolgáló-számítógépek közötti adatforgalmat.

**Mik azok az Active Directory tartományi szolgáltatások?
**Az Active Directory tartományi szolgáltatások (AD DS) címtáradatokat tárolnak, valamint a felhasználók és tartományok közötti kommunikációt vezérlik, beleértve a felhasználók bejelentkezési folyamatait, hitelesítését és címtárkereséseit. Az Active Directory tartományvezérlő az AD DS szolgáltatást futtató kiszolgáló. Az AD DS szolgáltatásról az [Active Directory tartományi szolgáltatások](http://technet.microsoft.com/en-us/library/cc770946(ws.10).aspx) című TechNet-cikkben talál bővebb információt.

**Mire használhatja a biztonsági rést a támadó?**
A biztonsági rést sikeresen kihasználó támadó hálózati erőforrásokhoz kaphat hozzáférést vagy kódot futtathat egy konkrét jogosult felhasználó jogosultságainak felhasználásával. Ha a felhasználó rendszergazdai jogosultságokkal rendelkezik, a támadó programokat telepíthet; adatokat tekinthet meg, módosíthat és törölhet; vagy korlátozás nélküli jogosultságokkal rendelkező új fiókokat hozhat létre.

**Hogyan használhatja ki a támadó a** **biztonsági rést?**
A biztonsági rés kihasználásához a támadónak először érvényes tartományi fiókhoz társított, visszavont tanúsítványt kell szereznie. Ezután úgy használhatja ki a biztonsági rést, ha a korábban visszavont tanúsítvánnyal hitelesíti magát az Active Directory tartományon, így hozzáférést szerez a hálózati erőforrásokhoz, vagy programkódot futtat a tanúsítványhoz társított, meghatározott jogosult felhasználó jogosultságainak felhasználásával.

**Mely rendszerekre jelenti a legnagyobb veszélyt a biztonsági rés?**
Ez a biztonsági rés érinti az összes olyan rendszert, amelyen az érintett operációs rendszerek futnak, és amelyekre telepítettek Active Directory szolgáltatásokat.

**Mi a frissítés feladata?**
A frissítés a biztonsági rés megszüntetése érdekében módosítja a tanúsítványok érvényességének CRL listán történő ellenőrzési módját az Active Directory szolgáltatásban.

**Ismert volt-e a nyilvánosság előtt ez a biztonsági rés a biztonsági közlemény kiadásának idején?**
Nem. A Microsoft ezt a célt szolgáló koordinált tájékoztatáson keresztül szerzett tudomást a biztonsági résről.

**A közlemény kibocsátásáig kapott-e a Microsoft a biztonsági rés kihasználásáról tájékoztató értesítést?**
Nem. A Microsoft nem kapott olyan információt, hogy a biztonsági közlemény eredeti változatának megjelenéséig a biztonsági rést felhasználták volna nyilvános támadásra.

-   **Fájlverziószám ellenőrzése**

    Mivel a Microsoft Windows operációs rendszernek több verziója és kiadása is van, a lépések az egyes rendszereken eltérőek lehetnek. Ilyen esetben a megfelelő lépések végrehajtásához az adott termék dokumentációja ad segítséget.

    1.  Kattintson a **Start** menü **Keresés** parancsára.
    2.  A **Keresés eredménye** ablak **Keresés** ablaktáblájában válassza a **Minden mappa és fájl** elemet.
    3.  **A fájlnév része vagy egésze** mezőbe írja be a táblázatban szereplő megfelelő fájl nevét, majd kattintson a **Keresés** gombra.
    4.  A fájlok listájában kattintson jobb gombbal a megfelelő fájl nevére, majd kattintson a **Tulajdonságok** parancsra.
        **Megjegyzés** Az operációs rendszer vagy a programok verziószámától függően elképzelhető, hogy a fájlinformációs táblázatban szereplő fájlok némelyike nem található meg a rendszerben.
    5.  A **Verzió** lapon hasonlítsa össze a rendszeren lévő fájl verziószámát a megfelelő táblázatban szereplő fájl verziószámával.
        **Megjegyzés** A telepítés során a fájl verziószámán kívül más attribútumok is megváltozhatnak. Ezen attribútumok fájlinformációs táblázatban szereplő adatokkal való összevetése a frissítés alkalmazásának nem ajánlott ellenőrzési módja. Előfordulhat az is, hogy a telepítés során bizonyos esetekben a rendszer fájlokat nevez át. Ha a fájl- vagy verzióadatok nem találhatók, ellenőrizze a frissítés telepített példányát egyéb módszerek segítségével.

-   **Beállításkulcsok ellenőrzése**

**A frissítés telepítésének ellenőrzése**

-   **Microsoft Baseline Security Analyzer**

    A Microsoft Baseline Security Analyzer (MBSA) eszközzel ellenőrizheti, hogy a biztonsági frissítés telepítve van-e az érintett számítógépen. További tájékoztatásért tekintse meg a közlemény korábbi részének **Észlelési és telepítési eszközök, útmutatás** című fejezetét.

-   **Fájlverziószám ellenőrzése**

    Mivel a Microsoft Windows operációs rendszernek több verziója és kiadása is van, a lépések az egyes rendszereken eltérőek lehetnek. Ilyen esetben a megfelelő lépések végrehajtásához az adott termék dokumentációja ad segítséget.

    1.  Kattintson a **Start** menü **Keresés** parancsára.
    2.  A **Keresés eredménye** ablak **Keresés** ablaktáblájában válassza a **Minden mappa és fájl** elemet.
    3.  **A fájlnév része vagy egésze** mezőbe írja be a táblázatban szereplő megfelelő fájl nevét, majd kattintson a **Keresés** gombra.
    4.  A fájlok listájában kattintson jobb gombbal a megfelelő fájl nevére, majd kattintson a **Tulajdonságok** parancsra.
        **Megjegyzés** Az operációs rendszer vagy a programok verziószámától függően elképzelhető, hogy a fájlinformációs táblázatban szereplő fájlok némelyike nem található meg a rendszerben.
    5.  A **Verzió** lapon hasonlítsa össze a rendszeren lévő fájl verziószámát a megfelelő táblázatban szereplő fájl verziószámával.
        **Megjegyzés** A telepítés során a fájl verziószámán kívül más attribútumok is megváltozhatnak. Ezen attribútumok fájlinformációs táblázatban szereplő adatokkal való összevetése a frissítés alkalmazásának nem ajánlott ellenőrzési módja. Előfordulhat az is, hogy a telepítés során bizonyos esetekben a rendszer fájlokat nevez át. Ha a fájl- vagy verzióadatok nem találhatók, ellenőrizze a frissítés telepített példányát egyéb módszerek segítségével.

-   **Beállításkulcsok ellenőrzése**

    A biztonsági frissítés által telepített fájlokat a **Hivatkozási táblázatban** feltüntetett rendszerleíróadatbázis-kulcsok segítségével is ellenőrizheti.

    Nem biztos, hogy ezek a rendszerleíró kulcsok az összes telepített fájlt tartalmazzák. Ne feledje azt sem, hogy ha a rendszergazda vagy az OEM gyártó beépíti a biztonsági frissítést a Windows rendszer telepítési forrásfájljaiba, a program esetleg hibásan hozza létre a rendszerleíró kulcsokat.

#### Tudnivalók a központi telepítésről
  
**A frissítés telepítése**
  
A biztonsági frissítés telepítésekor a telepítő ellenőrzi, hogy a számítógépen frissítendő fájlokat egy korábbi Windows-gyorsjavítás nem frissítette-e.
  
A közleményben alkalmazott szakkifejezésekkel (pl. *gyorsjavítás*) kapcsolatban tekintse meg a [Microsoft Tudásbázis 824684. számú cikkét](http://support.microsoft.com/kb/824684).
  
A biztonsági frissítés telepítője az alábbi kapcsolók használatát támogatja:
  
| Kapcsoló                             | Leírás                                                                                                                                                              |  
|--------------------------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------|  
| **/?, /h,** **/help**                | A támogatott kapcsolóknál megjeleníti a súgót.                                                                                                                      |  
| **/quiet**                           | Nem jeleníti meg az állapot- és hibaüzeneteket.                                                                                                                     |  
| **/norestart**                       | A **/quiet** kapcsolóval együtt alkalmazva a rendszer telepítés után nem indul újra, még akkor sem, ha a telepítés befejezéséhez a rendszert újra kellene indítani. |  
| **/warnrestart:&lt;seconds&gt;**     | Ha ezt **/quiet** elemmel kapcsolja össze, a telepítő figyelmezteti a felhasználót az újraindítás előtt.                                                            |  
| **/promptrestart**                   | Ha ezt **/quiet** elemmel kapcsolja össze, a telepítő figyelmeztetést küld az újraindítás előtt.                                                                    |  
| **/forcerestart**                    | Ha ezt **/quiet** elemmel kapcsolja össze, a telepítő kényszerített módon bezárja az alkalmazásokat, és újraindítja a rendszert.                                    |  
| **/log:&lt;file name&gt;**           | A megadott fájlba való bejelentkezést teszi lehetővé.                                                                                                               |  
| **/extract:&lt;destination&gt;**     | Kibontja a csomag tartalmát a célmappába.                                                                                                                           |  
| **/uninstall /kb:&lt;KB Number&gt;** | Eltávolítja a biztonsági frissítést.                                                                                                                                |
  
**Megjegyzés** A wusa.exe telepítő részletes leírását lásd a [Különböző módosítások a Windows 7 rendszerben](http://technet.microsoft.com/en-us/library/dd871148(ws.10).aspx) TechNet cikk Windows Update önálló telepítő c. részében.

    1.  Kattintson a **Start** elemre, majd a **Keresés indítása** mezőben adja meg a frissítőfájl nevét.  
    2.  Amikor a fájl megjelenik a **Programok** között, jobb egérgombbal kattintson a fájlnévre, majd a **Tulajdonságok** lehetőségre.

    4.  Az adatok összevetéséhez (pl. fájlverzió, a módosítás dátuma) a **Részletek** lapra is kattinthat, a közlemény Tudásbázis cikkében ismertetett fájlinformációs táblázatok alapján.  
        **Megjegyzés:** A telepítés során a fájl verziószámán kívül más attribútumok is megváltozhatnak.
