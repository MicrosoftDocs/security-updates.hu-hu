Security Bulletin

A Microsoft számú biztonsági közleménye MS08-040 - Fontos
=========================================================

A Microsoft SQL Server biztonsági rései jogok kiterjesztését okozhatják (941203)
--------------------------------------------------------------------------------

Közzétéve: 2008. július 8. | Frissítve: 2009. március 18.

**Verzió:** 1.9

### Általános tudnivalók

#### Összefoglalás

Ez a biztonsági frissítés négy közvetlenül jelzett biztonsági rést szüntet meg. A legsúlyosabb kockázattal járó biztonsági rés lehetővé teszi, hogy az azt kihasználó támadó kódot futtasson, és átvegye a teljes érintett rendszer irányítását. Ezt követően a hitelesített támadó programokat telepíthet, illetve adatokat tekinthet meg, módosíthat és törölhet, de teljes rendszergazda jogosultságokkal rendelkező új fiókokat is létrehozhat.

A biztonsági frissítés az SQL Server 7.0, SQL Server 2000, SQL Server 2005, Microsoft Data Engine (MSDE) 1.0, Microsoft SQL Server 2000 Desktop Engine (MSDE 2000), Microsoft SQL Server 2005 Express Edition, Microsoft SQL Server 2000 Desktop Engine (WMSDE), valamint a Windows Internal Database (WYukon) rendszer támogatott kiadásaira nézve fontos besorolású. További tudnivalókat a közlemény későbbi részében az **Érintett és nem érintett szoftverek** cím alatt talál.

A biztonsági rések megszüntetése végett a frissítés módosítja az oldalak újrafelhasználási módját az SQL Server rendszerben, több memóriát oszt ki a konvertáláshoz, és módosul a lemezfájlok feltöltése előtt és az utasítások beillesztése kapcsán végzett hitelesítés is. A biztonsági résekkel kapcsolatban bővebben az adott problémával foglakozó gyakran feltett kérdéseket ismertető szakaszban, **A biztonsági réssel kapcsolatos tudnivalók** című részben olvashat.

**Javaslat.** A Microsoft a biztonsági frissítés lehető leghamarabb történő telepítését javasolja ügyfeleinek.

**Ismert problémák** A [Microsoft Tudásbázis 941203. számú](http://support.microsoft.com/kb/941203) cikkében olvasható, hogy a felhasználók a biztonsági frissítés telepítésekor milyen ismert problémákkal találkozhatnak. A cikkben a problémákra javasolt megoldások is szerepelnek. A frissítések telepítéséről a további részleteket lásd a jelen fejezet **Gyakran ismételt kérdések a biztonsági frissítéssel kapcsolatban** alfejezetében.

#### Érintett és nem érintett szoftverek

Az itt felsorolt szoftvereket a verziók és változatok érintettségének felderítése érdekében teszteltük. A többi verzió, illetve kiadás támogatási életciklusa végére ért vagy nem érintett. Az egyes szoftverek és azok verzióinak támogatási életciklusáról a [Microsoft termékek terméktámogatási életciklusát ismertető webhelyen](http://go.microsoft.com/fwlink/?linkid=21742) talál felvilágosítást.

A szoftververziótól vagy -kiadástól függően választani kell az alábbi GDR és a QFE szoftverfrissítési hivatkozás között, amikor manuálisan telepíti a frissítést a Microsoft Letöltőközpontból. Arról, hogy melyik frissítést telepítse az adott rendszerre, további részletekért lásd a jelen fejezet **Gyakran ismételt kérdések a biztonsági frissítéssel kapcsolatban** alfejezetében.

<span></span>  
**Miért módosították a biztonsági közleményt 2008. augusztus 12-én?**    
A Microsoft biztonsági frissítés módosítása közli a SQL Server 2005 Service Pack 2 biztonsági frissítés új telepítőkódját, egyben elhárít egy telepítési problémát. A módosítás tehát csak a telepítőkódot érinti. A biztonsági frissítéshez kapcsolódó bináris fájlok nem változtak. Noha a WSUS rendszergazdák a frissítés új változataként érzékelik ezt, ha a felhasználó korábban már telepítette a biztonsági frissítést, nincs szükség újbóli telepítésre.
  
**Miért módosították a biztonsági közleményt 2008. július 23-án?**    
A frissített közlemény Microsoft SQL Server 2000 Desktop Engine (WMSDE) és Windows Internal Database (WYukon) eltávolításával foglalkozó része bővült annak érdekében, hogy felhívja a felhasználók figyelmét arra, hogy a WMSDE vagy a WYukon biztonsági frissítésének eltávolítása a WMSDE vagy a WYukon példányát teljesen törli a rendszerről.
  
**Miért módosították a közleményt 2008. július 18-án?**    
A közleményt egy tájékoztatóval frissítették, mely szerint a Windows Server Update Services (WSUS) Microsoft SQL Server 2000 Desktop Engine (WMSDE) által végzett frissítése során az észlelési mód megváltozott. A szóban forgó módosítás csak észlelési módosítás. A bináris fájlok nem változtak.
  
**Miért frissítették a közleményt 2008. július 11-én?**    
A közleményt egy tájékoztatóval frissítették, mely szerint a [Microsoft Tudásbázis kapcsolódó 948110. cikkének](http://support.microsoft.com/kb/948110)**A biztonsági frissítéssel kapcsolatos ismert problémák** című része az SQL Server 2000 Desktop Engine (WMSDE) termékváltozat-frissítéseit érintő észlelési probléma leírásával frissült.
  
**Az SQL verziómhoz GDR és QFE frissítés is megtalálható. Hogyan válasszam ki a megfelelő frissítést?**    
Nézze meg az SQL Server verziószámát. Az SQL Server verziószám meghatározásának részleteit lásd a [Microsoft Tudásbázis 321185 sz.](http://support.microsoft.com/kb/321185) cikkében.
  
Ezután az alábbi táblázatban keresse ki azt a tartományt, amelybe az SQL Server verziószám tartozik. Az ennek megfelelő frissítést kell telepítenie.
  
**Megjegyzés** Ha az SQL Server verziószám nem esik bele az alábbi táblázat egyik tartományába se, az SQL Server verzió már nem támogatott. Frissítsen a legújabb Service Pack csomagra vagy SQL Server termékre, és vegye használatba a jelenlegi és a jövőbeni biztonsági frissítéseket.

**Miért nem ajánlja fel a rendszer a QFE KB948108 szoftverfrissítést az SQL Server 2005 Service Pack 2 rendszeremhez?**  
Amint a [Microsoft Tudásbázis 935897 sz. cikkéből](http://support.microsoft.com/kb/935897) kitűnik, az SQL Server 2005 gyorsjavításai nem összesítő jellegűek. A gyorsjavítások tesztelési időszaka jellemzően rövidebb, mint az általános terjesztésű kiadásoké (GDR). Ha telepítette az áprilisi [7-es összesítő frissítőcsomagot az SQL Server 2005 Service Pack 2 rendszerhez](http://support.microsoft.com/kb/949095), illetve a júniusi [8-as összesítő frissítőcsomagot az SQL Server 2005 Service Pack 2 rendszerhez](http://support.microsoft.com/kb/951217), a rendszer nem ajánlja fel a QFE KB948108 szoftverfrissítést, mivel a QFE KB948108 frissítés elemei az áprilisi és a júniusi összesítő frissítésben is megtalálhatóak. A frissítés azonban vonatkozik azokra a felhasználókra, akik nem telepítették sem az áprilisi, sem a júniusi összesített frissítést SQL Server 2005 Service Pack 2 rendszerükhöz.

**Az SQL Server 2000 és MSDE 2000 rendszerre vonatkozó hivatkozási táblázatok** **Tudnivalók a biztonsági frissítésről című része jelzi, hogy melyik frissítésnél van esetleg szükség újraindításra. Honnan tudhatom meg, hogy az adott frissítés igényli-e a rendszer újraindítását?**  
SQL Server 2000 és MSDE 2000 esetében függő szolgáltatás vehető igénybe. A függő szolgáltatás nem indul újra, noha a fölérendelt szolgáltatást leállították és újraindították. Memóriaoldalak újrafelhasználási biztonsági rése – CVE-2008-0085
----------------------------------------------------------------

<span></span>
A memóriaoldalak újrafelhasználása az SQL Server programban adatokhoz való illetéktelen hozzáférést okozhat. Az adatbázis-kezelői hozzáférést szerző támadó a biztonsági rés kihasználásával hozzáférhet az ügyféladatokhoz.

A biztonsági rés standard bejegyzésének megtekintéséhez a normál biztonsági rés besorolási listáján lásd: [CVE-2008-0085](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2008-0085).

#### A memóriaoldalak újrafelhasználási biztonsági résének (CVE-2008-0085) hibáit enyhítő tényezők

Az enyhítő tényezők közé sorolható egy beállítás, konfiguráció vagy általánosan bevált gyakorlat, amely alapértelmezett állapotában csökkentheti a biztonsági rés kihasználásának valószínűségét. A következő enyhítő tényezők hasznosnak bizonyulhatnak:

-   A biztonsági rés kihasználásához a támadónak adatbázis-kezelői hozzáférésre van szüksége az adatbázishoz, naplókhoz, háttérmentett fájlokhoz, illetve a biztonsági mentések naplófájljaihoz.

#### Lehetséges megoldások a memóriaoldalak újrafelhasználási biztonsági réséhez (CVE-2008-0085)

A lehetséges megoldás olyan beállítást vagy konfigurációs módosítást takar, amely magát a biztonsági rést nem szünteti meg, azonban a frissítés telepítése előtt segít az ismert támadási lehetőségek kivédésében. A Microsoft alaposan megvizsgálta a következő lehetséges megoldásokat, és tájékoztatja a felhasználókat, amennyiben a megoldás csökkenti a működőképességet:

-   **A Common Criteria szabványnak való megfelelés engedélyezése az SQL 2005 Service Pack 2 rendszeren**

    A [Common Criteria szabványnak való megfelelés](http://msdn.microsoft.com/en-us/library/bb326650.aspx) engedélyezéséhez futtassa a következő parancsfált (rendszergazdaként lépjen be az SQL Server programba). A parancsfájl futtatását követően indítsa újra az SQL Server programot.

    `sp_configure 'show advanced options', 1;GORECONFIGURE;GOsp_configure 'common criteria compliance enabled', 1;GORECONFIGUREGO`

    **A megoldás hatása:** A Common Criteria szabványnak való megfelelés engedélyezése némileg csökkenti a rendszer teljesítményét.

    **A megoldás visszavonása.** A Common Criteria szabványnak való megfelelés letiltásához futtassa a következő parancsfált (rendszergazdaként lépjen be az SQL Server programba). A parancsfájl futtatását követően indítsa újra az SQL Server programot.

    `sp_configure 'show advanced options', 1;GORECONFIGURE;GOsp_configure 'common criteria compliance enabled', 0;GORECONFIGUREGO`

#### Gyakran feltett kérdések a memóriaoldalak újrafelhasználási biztonsági résével (CVE-2008-0085) kapcsolatban

**Mire terjed ki a biztonsági rés hatása?**  
A memóriaoldalak újrafelhasználása az SQL Server programban adatokhoz való illetéktelen hozzáférést okozhat. Az adatbázis-kezelői hozzáférést szerző támadó a biztonsági rés kihasználásával hozzáférhet olyan ügyféladatokhoz, melyek megtekintése egyébként le lenne tiltva számára.

**Mi okozza a biztonsági rést?**  
A biztonsági rés oka a memóriaoldalak újrafelhasználása az SQL Server programban. Az SQL Server a memória újrakiosztása során nem tudja inicializálni a memóriaoldalakat.

**Mire használhatja a biztonsági rést a támadó?**  
A támadó hozzáférhet másik felhasználó munkamenetére vonatkozó adatokat tartalmazó memóriaoldalak tartalmához.

**Hogyan használhatja ki a támadó a biztonsági rést?**  
Az adatbázis-kezelői hozzáféréssel rendelkező támadó hozzáférhet egy másik felhasználó munkamenetének nem inicializált memóriaoldalaihoz, ha biztonsági másolatot készít róluk egy általa felügyelt helyre.

**Mely rendszerekre jelenti a legnagyobb veszélyt a biztonsági rés?**  
Azok a rendszerek, ahol nem megbízható felhasználók használják az SQL Server programot.

**Mi a frissítés feladata? **
A frissítés a biztonsági rés megszüntetéséhez módosítja azt az eljárást, ahogy az SQL Server kezeli az oldalak újrafelhasználását.

**Hogyan törölhetem a maradék adatokat az adatbázisokból? **
A következő új, rendszerben tárolt eljárás célja az adatbázisoldalak szabad területén maradt adatok fizikai törlése olyan környezetben, ahol az adatok, illetve háttérmentésre szolgáló fájlok fizikai biztonsága kockázatnak van kitéve:

`sp_clean_db_free_space`

Ez a tárolt eljárás eltávolítja az SQL Server adatmódosítási eljárásai során az adatbázisoldalakon maradt információkat. Ez az eljárás minden fájl minden oldalát törli az adatbázisból.

`sp_clean_db_free_space@dbname = 'database_name',[ @cleaning_delay = 'delay_in_seconds' ]`

**Argumentumok**

`@dbname =  'database_name',`

-   A törlendő adatbázis neve. A *dbname* értéke **nvarchar(384)**, és nem lehet NULLA.

`[ @cleaning_delay = 'delay_in_seconds' ]`

-   Az IO rendszer hatásának redukálása végett késedelem adható meg az egyes oldalak törlése között. A *cleaning\_delay* értéke **int**, alapértelmezésben 0.

A következő példa minden maradék adatot töröl az AdventureWorks adatbázisból.

`USE MasterGOEXEC sp_clean_db_free_space@dbname = 'AdventureWorks';`

**Ismert volt-e a nyilvánosság előtt ez a biztonsági rés a biztonsági közlemény kiadásának idején? **
Nem. A Microsoft ezt a célt szolgáló tájékoztatáson keresztül szerzett tudomást a biztonsági résről. A Microsoft nem kapott olyan információt, hogy e közlemény eredeti változatának megjelenéséig a biztonsági rést a nyilvánosság tudomására hozták volna. A biztonsági frissítés a közvetlenül jelzett, valamint a belső vizsgálatok eredményeképpen feltárt egyéb hiányosságokat szünteti meg.

**A közlemény kibocsátásáig kapott-e a Microsoft a biztonsági rés kihasználásáról tájékoztató értesítést? **
Nem. E közlemény eredeti verziójának kiadásáig a Microsoft nem kapott olyan információt, amely arra utalt volna, hogy ezt a biztonsági rést nyilvános támadásra használták, és nem találkozott olyan nyilvánosságra hozott leírásokkal, amelyek tartalmazták volna egy sikeres támadás forgatókönyvét.

    Mivel a Microsoft Windows operációs rendszernek több változata van, a lépések az egyes rendszereken eltérőek lehetnek. Ilyen esetben a megfelelő lépések végrehajtásához az adott termék dokumentációja ad segítséget.

        **Megjegyzés** A telepítés során a fájl verziószámán kívül más attribútumok is megváltozhatnak. Előfordulhat az is, hogy a telepítés során bizonyos esetekben a rendszer fájlokat nevez át.
