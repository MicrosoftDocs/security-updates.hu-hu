Security Bulletin

A Microsoft számú biztonsági közleménye MS08-076 - Fontos
=========================================================

A Windows Media Components biztonsági rései távoli programkódfuttatást tesznek lehetővé (959807)
------------------------------------------------------------------------------------------------

Közzétéve: 2008. december 9. | Frissítve: 2009. november 24.

**Verzió:** 5.0

### Általános tudnivalók

#### Összefoglalás

Ez a biztonsági frissítés a a következő Windows Media összetevők két közvetlenül jelzett biztonsági rését szünteti meg: Windows Media Player, Windows Media Format Runtime és Windows Media Services. A legsúlyosabb besorolású biztonsági rés távolról történő kódfuttatást tehet lehetővé. Ha valamely felhasználó felügyeleti jogosultságokkal bejelentkezett a rendszerbe, akkor a biztonsági rést kihasználó támadó teljes mértékben átveheti az irányítást a számítógép felett, Ezt követően a támadó programokat telepíthet, illetve adatokat tekinthet meg, módosíthat és törölhet, de korlátozás nélküli jogosultságokkal rendelkező új fiókokat is létrehozhat. A kevesebb jogosultsággal bíró fiókkal rendelkező felhasználók esetében kisebb a veszélyeztetettség, mint a rendszergazdai jogosultságokkal rendelkezők esetén

A biztonsági frissítés Windows Media Player 6.4, Windows Media Format Runtime 7.1, Windows Media Format Runtime 9.0, Windows Media Format Runtime 9.5, Windows Media Format Runtime 11, Windows Media Services 4.1, Windows Media Services 9 Series és Windows Media Services 2008 rendszerek esetében Fontos besorolást kapott. További tudnivalókért tekintse meg az **Érintett és nem érintett szoftverek** című részt.

A frissítés az első biztonsági rés megszüntetéséhez módosítja a Windows Media hitelesítési válaszainak érvényesítési eljárását. A frissítés a második biztonsági rés megszüntetéséhez biztosítja, hogy a Windows Media ügyfelek az ISATAP-címeket használó kiszolgálókat külsőként kezeljék. A biztonsági résekkel kapcsolatban bővebben az adott problémával foglakozó gyakran feltett kérdéseket ismertető szakaszban, **A biztonsági réssel kapcsolatos tudnivalók** című részben olvashat.

**Javaslat.** A Microsoft a biztonsági frissítés lehető leghamarabb történő telepítését javasolja ügyfeleinek.

**Ismert problémák** Nincsenek

#### Érintett és nem érintett szoftverek

Az itt felsorolt szoftvereket a verziók és változatok érintettségének felderítése érdekében teszteltük. A többi verzió, illetve kiadás támogatási életciklusa végére ért vagy nem érintett. Az egyes szoftverek és azok verzióinak támogatási életciklusáról a [Microsoft termékek terméktámogatási életciklusát ismertető webhelyen](http://go.microsoft.com/fwlink/?linkid=21742) talál felvilágosítást.

**Miért adták ki újra a közleményt 2009. november 24-én?**  
A Microsoft az újonnan kiadott közleménnyel egy új frissítési csomagot ajánl a felhasználóknak, amely a Windows XP Embedded rendszerekhez alkalmazható. A frissítés célja a közleményben szerepelő eredeti Windows XP frissítőcsomagoknál tapasztalt telepítési és észlelési problémák megoldása.

A Microsoft megújította a Windows XP Service Pack 2 (KB952069) és a Windows XP Service Pack 3 (KB952069) létező frissítőcsomagjait annak érdekében, hogy azokat a program ne ajánlja fel a Windows XP Embedded rendszerekhez. A frissítést már telepített felhasználóknak - mivel már védettséget élveznek - nincs más teendőjük.

A Windows XP Embedded operációs rendszer felhasználóinak haladéktalanul telepíteniük kell a frissítést. A frissítések manuálisan is letölthetőek a [Windows XP Embedded Service Pack 2](http://www.microsoft.com/downloads/details.aspx?familyid=c8f59b79-7d09-4976-a678-9aef5975c186) (KB972187) vagy a [Windows XP Embedded Service Pack 3](http://www.microsoft.com/downloads/details.aspx?familyid=502dcf76-aec3-48bf-930d-0c02b1604cd1) (KB972187) rendszerhez. Az új kiadás más frissítéseket nem érint.

**A Windows XP Embedded rendszer bemutatása**  
A Microsoft Windows XP Embedded rendszer a Windows XP Professional operációs rendszer különleges változata, amely a Windows XP Professional funkcióinak gazdag skáláját a beágyazott szolgáltatásokra is kiterjeszti. A Windows XP Embedded rendszer a kiskereskedelmi szektor számára optimalizált változat, beépített hálózati funkciókkal és szabványos plug-and-play támogatással. A Windows XP Embedded rendszer számos periféria és készülék közös szolgáltatási ponthoz (POS) való csatlakoztatását teszi lehetővé. A Windows XP Embedded rendszerekkel kapcsolatban bővebben a [Microsoft Windows Embedded rendszerek fejlesztési központjában olvashat](http://msdn.microsoft.com/en-us/windowsembedded/default.aspx).

**Miért adták ki újra a közleményt 2009. január 13-án?**  
A Microsoft azért adta ki újra a közleményt, mert az új frissítéseket tartalmaz a Windows Media Format Runtime 9.5 programhoz a Windows XP Service Pack 2 (KB952069) és a Windows XP Service Pack 3 (KB952069) rendszeren. Az új frissítések a Windows Media Format Runtime 9.5 programmal rendelkező Windows XP rendszereken nem teljes telepítést okozó telepítési problémát orvosolja, amely akkor következik be, ha a felhasználó a különálló KB944110 számú frissítést a KB952069 számú frissítés előtt telepítette. Ezekre a felhasználókra továbbra is veszélyt jelent a biztonsági rés, ezért le kell tölteniük és telepíteniük kell az újra kiadott frissítéseket. Az új kiadás más frissítéseket nem érint. A Windows Media Player, Windows Media Format Runtime és Windows Media Services összes többi támogatott és érintett verziójának azon felhasználóinak, akik már sikeresen telepítették az eredeti biztonsági frissítéseket, nincs további teendőjük. Az újra kiadott biztonsági frissítések az összes csatornán elérhetők, amelyen az eredeti biztonsági frissítés, beleértve az Automatikus frissítések, a Windows Update és a Windows Server Update Services szolgáltatást.

A Microsoft a módosított közleményben a Windows Media Player 6.4 és a Windows Media Services 4.1 programot a Microsoft Windows 2000 Service Pack 4 összes verzióján érintettként tünteti fel (nem csak a Microsoft Windows 2000 Server Service Pack 4-en). Ugyanaz a frissítés érvényes; ehhez az operációs rendszerhez nincs új frissítés. Azok a felhasználók azonban, akik nem telepítették a felajánlott, Windows Media Player 6.4 esetén KB954600 számú, Windows Media Services 4.1 esetén pedig KB952068 számú frissítést, most tegyék meg.

**Hol találhatók a fájlinformációs adatok?**

SPN-biztonsági rés - CVE-2008-3009
----------------------------------

<span></span>
A Windows Media összetevőiben a hitelesítő adatok továbbadását lehetővé tévő biztonsági rés található, amelyet kihasználva a támadó a helyi felhasználóval vagy a Windows Media Services hitelesíto adatokkal rendelkező felhasználóval megegyező jogosultsággal programkódot futtathat. A biztonsági rést Windows Media összetevőinek Service Principal Name (SPN) alkalmazási módja okozza.

A biztonsági rés standard bejegyzésének megtekintéséhez a normál biztonsági rés besorolási listáján lásd: [CVE-2008-3009](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2008-3009).

#### Az SPN biztonsági résének (CVE-2008-3009) hibáit enyhítő tényezők

Az enyhítő tényezők közé sorolható egy beállítás, konfiguráció vagy általánosan bevált gyakorlat, amely alapértelmezett állapotában csökkentheti a biztonsági rés kihasználásának valószínűségét. A következő enyhítő tényezők hasznosnak bizonyulhatnak:

-   A Windows Media Services az összes Windows Server rendszeren opcionális összetevő, ezért alapértelmezés szerint nem telepített.
-   Webalapú támadás esetén a támadónak olyan webhelyet kell üzemeltetnie, amely egy, a biztonsági rés kihasználását lehetővé tevő weblapot tartalmaz. A támadónak nincs lehetősége arra, hogy a felhasználót a rosszindulatú kódot tartalmazó webhely megnyitására kényszerítse. Ehelyett a támadóknak saját webhelyükre kell csábítaniuk a felhasználókat, ezt pedig rendszerint egy kifejezetten ezt a célt szolgáló hivatkozás segítségével érik el. Miután a hivatkozásra kattintott a felhasználó, bizonyos műveletek végrehajtását kérő üzenetet kap. A támadás sikeréhez a felhasználónak mindenképpen végre kell hajtani a kért műveleteket.
-   Windows Server 2008 és Windows Server 2008 x64 alapú rendszerek esetén a Windows Media Format Runtime 11 csak a Desktop Experience funkció telepítése után szerepel a rendszeren.

#### Lehetséges megoldások az SPN biztonsági résének (CVE-2008-3009) kiküszöbölésére

A Microsoft egyelőre nem talált lehetséges megoldást a biztonsági rés megszüntetésére.

#### Gyakran feltett kérdések az SPN biztonsági résével (CVE-2008-3009) kapcsolatban

**Mire terjed ki a biztonsági rés hatása?**  
A biztonsági rés távolról történő kódfuttatást tehet lehetővé. A biztonsági rést sikeresen kihasználó támadó a bejelentkezett felhasználóval azonos jogosultsággal futtathat programokat. Ha valamely felhasználó felügyeleti jogosultságokkal bejelentkezett a rendszerbe, akkor a biztonsági rést kihasználó támadó teljes mértékben átveheti az irányítást a számítógép felett, Ezt követően a támadó programokat telepíthet, illetve adatokat tekinthet meg, módosíthat és törölhet, de korlátozás nélküli jogosultságokkal rendelkező új fiókokat is létrehozhat. A kevesebb jogosultsággal bíró fiókkal rendelkező felhasználók esetében kisebb a veszélyeztetettség, mint a rendszergazdai jogosultságokkal rendelkezők esetén  A biztonsági rést sikeresen kihasználó támadó Windows Media Services hitelesítő adatokkal is rosszindulatú kódot futtathat. Ebben az esetben a támadó a NETWORK SERVICE fiók jogosultságaival megegyező jogokkal rendelkezik, azokat a kódokat futtathatja..

**Mi okozza a biztonsági rést?**  
A Windows Media összetevői hibásan végzik az NTLM hitelesítő adatok védelmét annak megakadályozására, hogy a felhasználói hitelesítő adatokat ne lehessen visszatükrözni a felhasználó rendszerére, ezáltal lehetővé téve a felhasználói adatok illetéktelenek általi használatát.

**Mik az egyszerű szolgáltatásnevek (Service Principle Name - SPN)?**  
Az egyszerű szolgáltatásnév (SPN) az az egyedi név, amely alapján az ügyfél egy adott szolgáltatási példányt azonosít. Az SPN név az ügyfél számára segíti a szolgáltatás hitelesítését. Amikor az ügyfél csatlakozni kíván egy szolgáltatáshoz, megkeresi annak egy példányát, létrehoz a példányhoz tartozó SPN-nevet, csatlakozik a szolgáltatáshoz, majd a hitelesítéshez bemutatja az SPN-nevet. Az SPN-névvel kapcsolatban további információt itt talál: [Szolgáltatások és hitelesítés](http://msdn.microsoft.com/en-us/library/ms733130.aspx). Az SPN-nevek formátumával kapcsolatban további információkat az [egyedi SPN-nevek formátumát ismertető résznél](http://msdn.microsoft.com/en-us/library/ms677601(vs.85).aspx)talál.

**Mire használhatja a biztonsági rést a támadó?**  
A biztonsági rést kihasználó támadó a bejelentkezett felhasználóval egyező hozzáférést nyer a rendszerhez. Ha valamely felhasználó felügyeleti jogosultságokkal bejelentkezett a rendszerbe, akkor a biztonsági rést kihasználó támadó teljes mértékben átveheti az irányítást a számítógép felett, Ezt követően a támadó programokat telepíthet, illetve adatokat tekinthet meg, módosíthat és törölhet, de korlátozás nélküli jogosultságokkal rendelkező új fiókokat is létrehozhat. A kevesebb jogosultsággal bíró fiókkal rendelkező felhasználók esetében kisebb a veszélyeztetettség, mint a rendszergazdai jogosultságokkal rendelkezők esetén

**Hogyan használhatja ki a támadó a biztonsági rést?**  
A biztonsági rés akkor használható ki, ha a felhasználó rosszindulatú kiszolgálóhoz fér hozzá az Windows Media érintett verziójával. A támadó helyet adhat egy rosszindulatú kiszolgáló-megosztásnak vagy weboldalnak, ahol hitelesítés szükséges. Amikor az ügyfél csatlakozik a rosszindulatú kiszolgálóhoz, és elvégzi a hitelesítést, a támadó felhasználhatja az így kinyert hitelesítő adatokat, azokat más kiszolgálókra is továbbíthatja, ezáltal az ügyfélnek kiadva magát. A támadónak nincs lehetősége arra, hogy a felhasználót a különlegesen kialakított kiszolgálómegosztás vagy webhely megnyitására kényszerítse. Ehelyett a támadóknak az adott kiszolgálómegosztásra vagy webhelyre kell csábítaniuk a felhasználókat. Ezt rendszerint úgy érik el, hogy ráveszik őket arra, hogy kattintsanak rá egy e-mailben lévő hivatkozásra vagy azonnali üzenetkezelőben megjelenő kérésre.

**Mely rendszerekre jelenti a legnagyobb veszélyt a biztonsági rés?**  
A Windows Media Player, Windows Media Format Runtime és a Windows Media Services bármely érintett verzióját futtató rendszer fenyegetett. Ennek következtében azok a rendszerek a leginkább veszélyeztetettek a támadással szemben, amelyeken ezeket az összetevőket gyakran használják, így például a munkaállomások vagy terminálkiszolgálók.

**Mi a frissítés feladata?**  
A biztonsági frissítés azáltal oldja meg a biztonsági rés okozta problémát, hogy a hitelesítő adatok ismételt felhasználásának megelőzése érdekében módosítja a Windows Media Component hitelesítési válaszok érvényesítési módját.

**Ismert volt-e a nyilvánosság előtt ez a biztonsági rés a biztonsági közlemény kiadásának idején?**  
Nem. A Microsoft ezt a célt szolgáló tájékoztatáson keresztül szerzett tudomást a biztonsági résről. A Microsoft nem kapott olyan információt, hogy e közlemény eredeti változatának megjelenéséig a biztonsági rést a nyilvánosság tudomására hozták volna.

ISATAP biztonsági rés - CVE-2008-3010
-------------------------------------

<span></span>
A Windows Media támogatott verziójú összetevőiben adatokhoz való illetéktelen hozzáférést lehetővé tévő biztonsági rés található, amely az NTLM-hitelesítő adatok illetéktelenekhez való kerülését okozhatja. Valamennyi Windows Media-összetevő, amely ISATAP címet használó URL-hez csatlakozik, az NTLM-adatokat kiszivárogtathatja az URL-nek helyet adó kiszolgálónak. Így az intranet zónán kívüli támadó vállalati környezet NTLM hitelesítő adatait is megszerezheti.

A biztonsági rés standard bejegyzésének megtekintéséhez a normál biztonsági rés besorolási listáján lásd: [CVE-2008-3010](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2008-3010).

#### Az ISATAP biztonsági résének (CVE-2008-3010) hibáit enyhítő tényezők

Az enyhítő tényezők közé sorolható egy beállítás, konfiguráció vagy általánosan bevált gyakorlat, amely alapértelmezett állapotában csökkentheti a biztonsági rés kihasználásának valószínűségét. A következő enyhítő tényezők hasznosnak bizonyulhatnak:

-   A tűzfalakra vonatkozó gyakorlati tanácsok, valamint a szabványos alapértelmezett tűzfal-konfigurációk segítséget nyújtanak a hálózati erőforrásokat érő, a szervezet határain kívülről eredő támadások kivédésében. Gyakorlati tanácsként azt javasoljuk, hogy az internetkapcsolattal rendelkező számítógépeken a lehető legkevesebb port és protokoll legyen megnyitva.
-   Webalapú támadás esetén a támadónak olyan webhelyet kell üzemeltetnie, amely egy, a biztonsági rés kihasználását lehetővé tevő weblapot tartalmaz. A támadónak nincs lehetősége arra, hogy a felhasználót a rosszindulatú kódot tartalmazó webhely megnyitására kényszerítse. Ehelyett a támadóknak saját webhelyükre kell csábítaniuk a felhasználókat, ezt pedig rendszerint egy kifejezetten ezt a célt szolgáló hivatkozás segítségével érik el. Miután a hivatkozásra kattintott a felhasználó, bizonyos műveletek végrehajtását kérő üzenetet kap. A támadás sikeréhez a felhasználónak mindenképpen végre kell hajtani a kért műveleteket.
-   Windows Server 2008 és Windows Server 2008 x64 alapú rendszerek esetén a Windows Media Format Runtime 11 csak a Desktop Experience funkció telepítése után szerepel a rendszeren.

#### Lehetséges megoldások az ISATAP biztonsági résének (CVE-2008-3010) kiküszöbölésére

A lehetséges megoldás olyan beállítást vagy konfigurációs módosítást takar, amely magát a biztonsági rést nem szünteti meg, azonban a frissítés telepítése előtt segít az ismert támadási lehetőségek kivédésében. A Microsoft alaposan megvizsgálta a következő lehetséges megoldásokat, és tájékoztatja a felhasználókat, amennyiben a megoldás csökkenti a működőképességet:

-   **Tiltsa le a tűzfalon a 41-es protokolltípust (ISATAP)**

    Az ISATAP protokoll 41-es típusú protokollként szerepel az [RFC4214: Intra-Site Automatic Tunnel Addressing Protocol (ISATAP)](http://www.ietf.org/rfc/rfc4214.txt) részben. A protokolltípus letiltása megakadályozza, hogy a tűzfal mögötti rendszereken a támadók a biztonsági rést kihasználják. A Microsoft azt javasolja, hogy az internet felől bejövő összes kéretlen adatforgalmat tiltsa le, ezzel megakadályozhatja az esetleg más portokon vagy protokollokon keresztül történő támadásokat.

#### Gyakran feltett kérdések az ISATAP biztonsági résével (CVE-2008-3010) kapcsolatban

**Mire terjed ki a biztonsági rés hatása?**  
A biztonsági rés illetéktelen adatelérést okoz. A biztonsági rést kihasználó támadó megszerezheti a bejelentkezett felhasználó NTLM-hitelesítő adatait. Ezeket az adatokat ezt követően továbbíthatja a felhasználó rendszerére, ahol a bejelentkezett felhasználóval megegyező jogokkal hajthat végre programokat.

**Mi okozza a biztonsági rést?**  
Az érintett Windows Media-összetevők az internet-erőforrásokat helytelenül a helyi intranet részeként értelmezik, amennyiben ISATAP-címet használó kiszolgálóhoz csatlakoznak.

**Mi az ISATAP?**  
Az Intra-Site Automatic Tunnel Addressing Protocol (ISATAP) IPv6 kapcsolatot biztosít IPv4 intraneten belül. Az ISATAP RFC-elemét az Internet Engineering Task Force (IETF) tárolja. Az ISATAP protokollról bővebb információk az [RFC4214: Intra-Site Automatic Tunnel Addressing Protocol (ISATAP)](http://www.ietf.org/rfc/rfc4214.txt) részben.

**Mi az az NTLM?**  
Az NTLM-hitelesítés kérdés-válasz alapon működik.

**Mire használhatja a biztonsági rést a támadó?**  
A biztonsági rést sikeresen kihasználó támadó megszerezheti a felhasználó NTLM hitelesítő adatait.  Ezeket az adatokat ezt követően továbbíthatja a felhasználó rendszerére, ahol a bejelentkezett felhasználóval megegyező jogokkal hajthat végre programokat.

**Hogyan használhatja ki a támadó a biztonsági rést?**  
A biztonsági rés akkor használható ki, ha a felhasználó rosszindulatú kiszolgálóhoz fér hozzá az Windows Media érintett verziójával. A támadó helyet adhat egy rosszindulatú kiszolgáló-megosztásnak vagy weboldalnak, ahol hitelesítés szükséges. Amikor az ügyfél csatlakozik a rosszindulatú kiszolgálóhoz, és elvégzi a hitelesítést, a támadó felhasználhatja az így kinyert hitelesítő adatokat, azokat más kiszolgálókra is továbbíthatja, ezáltal az ügyfélnek kiadva magát. A támadónak nincs lehetősége arra, hogy a felhasználót a különlegesen kialakított kiszolgálómegosztás vagy webhely megnyitására kényszerítse. Ehelyett a támadóknak az adott kiszolgálómegosztásra vagy webhelyre kell csábítaniuk a felhasználókat. Ezt rendszerint úgy érik el, hogy ráveszik őket arra, hogy kattintsanak rá egy e-mailben lévő hivatkozásra vagy azonnali üzenetkezelőben megjelenő kérésre.

**Mely rendszerekre jelenti a legnagyobb veszélyt a biztonsági rés?**  
A Windows Media Player, Windows Media Format Runtime és a Windows Media Services bármely érintett verzióját futtató rendszer fenyegetett. Ennek következtében azok a rendszerek a leginkább veszélyeztetettek a támadással szemben, amelyeken ezeket az összetevőket gyakran használják, így például a munkaállomások vagy terminálkiszolgálók.

**Mi a frissítés feladata?**  
A frissítés a biztonsági rés megszüntetéséhez biztosítja, hogy a Windows Media Component ügyfelek az ISATAP címeket használó kiszolgálókat külsőként kezeljék, és ne adjanak ki felhasználói hitelesítő adatokat.

**A frissítés telepítésének ellenőrzése**

-   **Microsoft Baseline Security Analyzer**

    A Microsoft Baseline Security Analyzer (MBSA) eszközzel ellenőrizheti, hogy a biztonsági frissítés telepítve van-e az érintett számítógépen. További tájékoztatásért tekintse meg a közlemény korábbi részének **Észlelési és telepítési eszközök, útmutatás** című fejezetét.

-   **Fájlverziószám ellenőrzése**

    Mivel a Microsoft Windows operációs rendszernek több verziója és kiadása is van, a lépések az egyes rendszereken eltérőek lehetnek. Ilyen esetben a megfelelő lépések végrehajtásához az adott termék dokumentációja ad segítséget.

    1.  Kattintson a **Start** menü **Keresés** parancsára.
    2.  A **Keresés eredménye** ablak **Keresés** ablaktáblájában válassza a **Minden mappa és fájl** elemet.
    3.  **A fájlnév része vagy egésze** mezőbe írja be a táblázatban szereplő megfelelő fájl nevét, majd kattintson a **Keresés** gombra.
    4.  A fájlok listájában kattintson jobb gombbal a megfelelő fájl nevére, majd kattintson a **Tulajdonságok** parancsra.
        **Megjegyzés** Az operációs rendszer vagy a programok verziószámától függően elképzelhető, hogy a fájlinformációs táblázatban szereplő fájlok némelyike nem található meg a rendszerben.
    5.  A **Verzió** lapon hasonlítsa össze a rendszeren lévő fájl verziószámát a megfelelő táblázatban szereplő fájl verziószámával.
        **Megjegyzés** A telepítés során a fájl verziószámán kívül más attribútumok is megváltozhatnak. Ezen attribútumok fájlinformációs táblázatban szereplő adatokkal való összevetése a frissítés alkalmazásának nem ajánlott ellenőrzési módja. Előfordulhat az is, hogy a telepítés során bizonyos esetekben a rendszer fájlokat nevez át. Ha a fájl- vagy verzióadatok nem találhatók, ellenőrizze a frissítés telepített példányát egyéb módszerek segítségével.

-   **Rendszerleíró kulcsok ellenőrzése**

    A biztonsági frissítés által telepített fájlokat a **Hivatkozási táblázatban** feltüntetett rendszerleíróadatbázis-kulcsok segítségével is ellenőrizheti.

    Nem biztos, hogy ezek a rendszerleíró kulcsok az összes telepített fájlt tartalmazzák. Ne feledje azt sem, hogy ha a rendszergazda vagy az OEM gyártó beépíti a biztonsági frissítést a Windows rendszer telepítési forrásfájljaiba, a program esetleg hibásan hozza létre a rendszerleíró kulcsokat.

-   **Fájlverziószám ellenőrzése**

    Mivel a Microsoft Windows operációs rendszernek több verziója és kiadása is van, a lépések az egyes rendszereken eltérőek lehetnek. Ilyen esetben a megfelelő lépések végrehajtásához az adott termék dokumentációja ad segítséget.

    1.  Kattintson a **Start** menü **Keresés** parancsára.
    2.  A **Keresés eredménye** ablak **Keresés** ablaktáblájában válassza a **Minden mappa és fájl** elemet.
    3.  **A fájlnév része vagy egésze** mezőbe írja be a táblázatban szereplő megfelelő fájl nevét, majd kattintson a **Keresés** gombra.
    4.  A fájlok listájában kattintson jobb gombbal a megfelelő fájl nevére, majd kattintson a **Tulajdonságok** parancsra.
        **Megjegyzés** Az operációs rendszer vagy a programok verziószámától függően elképzelhető, hogy a fájlinformációs táblázatban szereplő fájlok némelyike nem található meg a rendszerben.
    5.  A **Verzió** lapon hasonlítsa össze a rendszeren lévő fájl verziószámát a megfelelő táblázatban szereplő fájl verziószámával.

Előfordulhat az is, hogy a telepítés során bizonyos esetekben a rendszer fájlokat nevez át. Ha a fájl- vagy verzióadatok nem találhatók, ellenőrizze a frissítés telepített példányát egyéb módszerek segítségével.

-   **Rendszerleíró kulcsok ellenőrzése**

#### Tudnivalók a központi telepítésről

**A frissítés telepítése**

A biztonsági frissítés telepítésekor a telepítő ellenőrzi, hogy a számítógépen frissítendő fájlokat egy korábbi Microsoft gyorsjavítás nem frissítette-e.

Ha a fájlok valamelyikét korábban már frissítette egy gyorsjavítással, a telepítő az SP1QFE vagy SP2QFE fájlokat másolja a rendszerre. Ellenkező esetben a telepítő az SP1GDR vagy SP2GDR fájlokat másolja a rendszerre. A biztonsági frissítések esetleg nem tartalmazzák a fájlok összes változatát. Erről a jelenségről további információt talál a [Microsoft Tudásbázis 824994](http://support.microsoft.com/kb/824994) számú cikkében.

A telepítőről további ismertetést a [Microsoft TechNet webhelyén](http://go.microsoft.com/fwlink/?linkid=38951) talál.
