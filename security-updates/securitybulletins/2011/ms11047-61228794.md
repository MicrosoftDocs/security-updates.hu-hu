A Microsoft számú biztonsági közleménye MS11-047 - Fontos
=========================================================

A Hyper-V szolgáltatásmegtagadást lehetővé tévő biztonsági rése (2525835)
-------------------------------------------------------------------------

Közzétéve: 2011. június 14.

**Verzió:** 1.0

### Általános tudnivalók

#### Összefoglalás

A biztonsági frissítés a közvetlenül jelentett biztonsági rés hibáit oldja meg a Windows Server 2008 Hyper-V és a Windows Server 2008 R2 Hyper-V használatával működő kiszolgálókon. A biztonsági rés szolgáltatásmegtagadást tehet lehetővé, ha a Hyper-V kiszolgálón vendégként futó virtuális gépek valamelyikén hitelesített felhasználó speciálisan kialakított csomagot küld a VMBus-ra. A biztonsági rés kihasználásához a támadónak rendelkeznie kell érvényes bejelentkezési adatokkal és képesnek kell lennie speciálisan kialakított tartalom küldésére valamely vendég virtuális gépről. Névtelen felhasználók nem tudják távolról kiaknázni ezt a biztonsági rést.

A biztonsági frissítés a Windows 2008 Server és a Windows Server 2008 R2 összes támogatott x64 alapú kiadására nézve fontos besorolásúnak minősül. További tudnivalókért lásd az **Érintett és nem érintett szoftverek** c. részt.

A biztonsági frissítés a biztonsági rés megszüntetése érdekében kijavítja a virtuális gépekről küldött hibás kérelmek Hyper-V Server általi feldolgozási módját. A biztonsági réssel kapcsolatban bővebben az adott problémával foglakozó gyakran feltett kérdéseket ismertető szakaszban, **A biztonsági réssel kapcsolatos tudnivalók** című részben olvashat.

**Javaslat** A felhasználók nagy részénél be van kapcsolva az automatikus frissítés; ilyenkor nincs más teendő, mert a rendszer automatikusan letölti és telepíti a biztonsági frissítést. Ha nem használ automatikus frissítést, keresse meg a vonatkozó frissítést és telepítse manuálisan. Az automatikus frissítés további beállításairól lásd a [Microsoft Tudásbázis 294871. cikkét](http://support.microsoft.com/kb/294871).

Ha a rendszergazda, vállalati telepítő vagy végfelhasználó manuálisan szeretné telepíteni a biztonsági frissítést, a Microsoft a minél előbbi telepítést javasolja frissítéskezelő szoftver vagy a [Microsoft Update](http://go.microsoft.com/fwlink/?linkid=40747) szolgáltatás igénybevételével.

Lásd még a közlemény **Észlelési és telepítési eszközök, útmutatás** fejezetét.

Gyakran ismételt kérdések a biztonsági frissítéssel kapcsolatban
----------------------------------------------------------------

**A rendszeremen nincs engedélyezve a Hyper-V. Telepíthetem a frissítést?**  
Nem. A frissítés csak akkor telepíthető az érintett operációs rendszerekre, ha a Hyper-V szerep engedélyezve van. A Hyper-V szerepet nem engedélyező rendszereket nem érinti a biztonsági rés.

**Felajánlják ezt a frissítést akkor is, ha nálam nincs engedélyezve a Hyper-V?**  
Az Automatikus frissítés során csak akkor kínálják fel, ha a Hyper-V szerep engedélyezve van. Ez mindenképpen igaz, akár a Kiszolgálómag telepítési opcióval telepítették az érintett operációs rendszert, akár nem.

**Kell-e ezt a frissítést a Hyper-V vendég virtuális gépekre is alkalmazni?**  
Ezt a frissítést a Hyper-V vendég virtuális gépekre nem kell alkalmazni, csak azokra a kiszolgálókra kell, amelyiken a Hyper-V szerep engedélyezve van. Az automatikus frissítés során csak a Hyper-V kiszolgálórendszerekre történik felajánlás, a virtualizált vendéggépeken belül nem.

**Mi a Hyper-V Server, és azt is érinti ez a biztonsági rés?**  
A Microsoft Hyper-V Server olyan önálló termék, amelyik a megbízható és optimalizált virtualizálási megoldásával lehetővé teszi, hogy a szervezetek jobban kihasználják a kiszolgálóikat és csökkentsék a költségeiket. Mivel a Hyper-V Server olyan önálló terméknek szánt, amelyik csak a Windows Hypervisor, a Windows Server illesztőmodelljét és virtualizálási összetevőit tartalmazza, kis ráfordítást és minimális terhelést jelent.

Ez a biztonsági rés a Microsoft Hyper-V Server 2008, Microsoft Hyper-V Server 2008 Service Pack 2 és Microsoft Hyper-V Server 2008 R2 termékeket érinti. A biztonsági rés javítása érdekében alkalmazni kell a megfelelő Windows Server 2008 és Windows Server 2008 R2 frissítéseket. Akiknél engedélyezve van az automatikus frissítés, automatikusan ajánlást kapnak ezen frissítések telepítésére.

A VMBus állandó szolgáltatásmegtagadást eredményező biztonsági rése - CVE-2011-1872  
-----------------------------------------------------------------------------------
  
A szolgáltatásmegtagadásban a Windows Server 2008 és a Windows Server 2008 R2 rendszeren futó Hyper-V érintett. A biztonsági rést az okozza, hogy a Hyper-V kiszolgálók nem megfelelően érvényesítik a gépi utasítások bizonyos sorozatait. A biztonsági rést sikeresen kihasználó támadó az érintett Hyper-V rendszert a válaszadások megszakítására kényszerítheti. Ez érintheti az adott rendszeren futó összes virtuális gépet.
  
A biztonsági rés standard bejegyzésének megtekintéséhez a normál biztonsági rés besorolási listáján lásd: [CVE-2011-1872](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2011-1872).
  
#### A VMBus állandó szolgáltatásmegtagadást eredményező biztonsági résének (CVE-2011-1872) súlyosságát enyhítő tényezők
  
Az enyhítő tényezők közé sorolható egy beállítás, konfiguráció vagy általánosan bevált gyakorlat, amely alapértelmezett állapotában csökkentheti a biztonsági rés kihasználásának valószínűségét. A következő enyhítő tényezők hasznosnak bizonyulhatnak:
  
-   A támadónak a biztonsági rés kihasználásához érvényes bejelentkezési adatokkal kell rendelkeznie, és képesnek kell lennie helyileg bejelentkezni a rendszerbe. Névtelen felhasználók nem tudják távolról kiaknázni ezt a biztonsági rést.
  
#### A VMBus állandó szolgáltatásmegtagadást eredményező biztonsági résének (CVE-2011-1872) lehetséges megoldásai
  
A Microsoft egyelőre nem talált lehetséges megoldást a biztonsági rés megszüntetésére.
  
#### Gyakran ismételt kérdések a VMBus szolgáltatásmegtagadást eredményező biztonsági résével (CVE-2011-1872) kapcsolatban
  
**Mire terjed ki a biztonsági rés hatása?**    
Ez a biztonsági rés szolgáltatásmegtagadást okoz. A biztonsági rést kihasználó támadó az érintett Hyper-V kiszolgálót a válaszadások megszakítására kényszerítheti, és a kiszolgálót újra kell indítani. Fontos megjegyezni, hogy a szolgáltatásmegtagadást okozó biztonsági rések a támadó számára nem teszik lehetővé programkód futtatását vagy jogosultságok illetéktelen megszerzését a Hyper-V kiszolgálón, de segítségükkel megakadályozható a rendszerben a kérelmek fogadása.
  
**Mi okozza a biztonsági rést?**    
A biztonsági rést a vendégről gazdagépre küldött egyes csomagok elégtelen ellenőrzése okozza. Az érvényesítés ezen hiányossága következtében ezek az utasítások az erőforrások túlzott felhasználását idézhetik elő a Hyper-V kiszolgáló alkalmazásban.
  
**Mi a Hyper-V?**    
A Hyper-V olyan hipervizor alapú technológia, amelyik a Windows Server 2008 és a Windows Server 2008 R2 kulcsfontosságú jellemzője. Ez biztosítja a méretezhető, megbízható és stabilan rendelkezésre álló virtualizálási platformot. Ez a környezet azért hasznos, mert virtuális gépeket hozhat létre és kezelhet, amelyek lehetővé teszik, hogy fizikailag egyetlen számítógépen több operációs rendszert futtasson, és elszigetelje egymástól az operációs rendszereket.
  
**Mire használhatja a biztonsági rést a támadó?**    
A biztonsági rést sikeresen kihasználó támadó a válaszadások megszakítására és újraindulásra kényszerítheti a felhasználó rendszerét. Ügyeljen arra, hogy a biztonsági rés hatására a Hyper-V kiszolgáló, beleértve az adott kiszolgálón lévő minden virtuális vendéggépet is, minden erőforrást lefoglalhat.
  
**Hogyan használhatja ki a támadó a biztonsági rést?**    
A támadónak a Hyper-V kiszolgálón lévő virtuális vendéggépek valamelyikén hitelesített felhasználónak kell lenni. A támadónak rendelkeznie kell azzal a lehetőséggel, hogy a rendszeren tetszőleges kódot hajthasson végre, például nem megbízható végrehajtható állomány futtatásával. Ez a végrehajtható fájl rosszindulatúan kialakított gépi utasítások sorozatát hívhatja meg, amelyek az erőforrások lefoglalását idézik elő, és a Hyper-V kiszolgálón így létrejött végtelen ciklus megakadályozza a más virtuális gépekről érkező kérelmek feldolgozását.
  
**Milyen jogosultsággal kell rendelkeznie a támadónak a szolgáltatásmegtagadás végrehajtásához?**    
A Hyper-V kiszolgáló és az adott kiszolgálón futó minden virtuális gép elleni szolgáltatásmegtagadási támadáshoz a támadónak rendszergazdai jogosultságokkal kell rendelkeznie a Hyper-V kiszolgálón lévő egyik virtuális gépen. Magához a Hyper-V kiszolgálóhoz nem szükséges rendszergazdai hozzáférés.
  
**Mely rendszerekre jelenti a legnagyobb veszélyt a biztonsági rés?**    
A biztonsági rés különösen azokra a Hyper-V kiszolgálókra veszélyes, amelyek olyan virtuális vendéggépeket működtetnek, amelyeket nem megbízható felhasználók is elérhetnek.
  
**Mi a frissítés feladata?**    
A frissítés kijavítja a virtuális gépekről küldött hibás kérelmek Hyper-V Server általi feldolgozási módját. A javítás lehetővé teszi, hogy a Hyper-V kiszolgálóalkalmazás megfelelően érvényesítse a vendégről gazdagépre küldött VMBus csomagokat.
  
**Ismert volt-e a nyilvánosság előtt ez a biztonsági rés az említett közlemény kiadásának idején?**    
Nem. A Microsoft ezt a célt szolgáló koordinált tájékoztatáson keresztül szerzett tudomást a biztonsági résről.
  
**A közlemény kibocsátásáig kapott-e a Microsoft a biztonsági rés kihasználásáról tájékoztató értesítést?**    
Nem. A Microsoft nem kapott olyan információt, hogy a biztonsági közlemény eredeti változatának megjelenéséig a biztonsági rést felhasználták volna nyilvános támadásra.

    3.  Az **Általános** lapon hasonlítsa össze a fájlméretet a közlemény Tudásbázis cikkében ismertetett fájlinformációs táblázatokkal.  
        **Megjegyzés:** Az operációs rendszer kiadásától vagy a telepített programok verziószámától függően elképzelhető, hogy a fájlinformációs táblázatban szereplő fájlok némelyike nem található meg a rendszerben.  
    4.  Az adatok összevetéséhez (pl. fájlverzió, a módosítás dátuma) a **Részletek** lapra is kattinthat, a közlemény Tudásbázis cikkében ismertetett fájlinformációs táblázatok alapján.  
        **Megjegyzés:** A telepítés során a fájl verziószámán kívül más attribútumok is megváltozhatnak. Ezen attribútumok fájlinformációs táblázatban szereplő adatokkal való összevetése a frissítés alkalmazásának nem ajánlott ellenőrzési módja. Előfordulhat az is, hogy a telepítés során bizonyos esetekben a rendszer fájlokat nevez át. Ha a fájl- vagy verzióadatok nem találhatók, ellenőrizze a frissítés telepített példányát egyéb módszerek segítségével.  
    5.  Kattinthat a **korábbi verziókat** felsoroló lapra is, majd összehasonlíthatja a korábbi fájlverzió adatait az aktuális, frissített fájléval.
