A Microsoft számú biztonsági közleménye MS10-041 - Fontos
=========================================================

A Microsoft .NET Framework biztonsági rése beavatkozást tehet lehetővé (981343)
-------------------------------------------------------------------------------

Közzétéve: 2010. június 8. | Frissítve: 2010. június 22.

**Verzió:** 1.2

### Általános tudnivalók

#### Összefoglalás

A biztonsági frissítés a Microsoft .NET Framework nyilvánosan jelentett biztonsági rését szünteti meg. A biztonsági rés aláírt XML-tartalmak észrevétlen adatmanipulálását teheti lehetővé. Egyedi alkalmazások esetén a biztonsági hatás az aláírt tartalom felhasználási módjától függ az adott egyedi alkalmazásban. A biztonsági rés nem érinti azokat az eseteket, amikor aláírt XML-üzeneteket továbbítanak biztonságos csatornán (pl. SSL) keresztül.

Ez a biztonsági frissítés a Microsoft Windows 2000 rendszeren futó Microsoft .NET Framework, a Windows XP, a Windows Server 2003, a Windows Vista, a Windows Server 2008, a Windows 7 és a Windows Server 2008 R2 összes támogatott kiadását tekintve Fontos besorolású. További információkért lásd az **Érintett és nem érintett szoftverek** alfejezetet.

A biztonsági frissítés a biztonsági rés megszüntetése érdekében módosítja az XMLDsig ajánlás végrehajtását a Microsoft .NET Framework rendszerben. A biztonsági résről **bővebben az adott problémával foglakozó, a gyakran feltett kérdéseket ismertető szakaszban, A biztonsági réssel kapcsolatos tudnivalók** című részben olvashat.

**Javaslat.**  A felhasználók nagy részénél be van kapcsolva az automatikus frissítés; ilyenkor nincs más teendő, mert a rendszer automatikusan letölti és telepíti a biztonsági frissítést. Ha nem használ automatikus frissítést, keresse meg a vonatkozó frissítést és telepítse manuálisan. Az automatikus frissítés további beállításairól lásd a [Microsoft Tudásbázis 294871. cikkét](http://support.microsoft.com/kb/294871).

Ha a rendszergazda, vállalati telepítő vagy végfelhasználó manuálisan szeretné telepíteni a biztonsági frissítést, a Microsoft a minél előbbi telepítést javasolja frissítéskezelő szoftver vagy a [Microsoft Update](http://go.microsoft.com/fwlink/?linkid=40747) szolgáltatás igénybevételével.

Lásd még a közlemény **Észlelési és telepítési eszközök, útmutatás** fejezetét.

**Ismert problémák** A[Microsoft Tudásbázis 981343. számú cikke](http://support.microsoft.com/kb/981343) ismerteti, hogy a felhasználók a biztonsági frissítés telepítésekor milyen ismert problémákkal találkozhatnak. A cikkben a problémákra javasolt megoldások is szerepelnek.

#### Érintett és nem érintett szoftverek

Az itt felsorolt szoftvereket a verziók és változatok érintettségének felderítése érdekében teszteltük. A többi verzió, illetve kiadás támogatási életciklusa végére ért vagy nem érintett. Az egyes szoftverek és azok verzióinak támogatási életciklusáról a [Microsoft termékek terméktámogatási életciklusát ismertető webhelyen](http://go.microsoft.com/fwlink/?linkid=21742) talál felvilágosítást.

Gyakran ismételt kérdések a biztonsági frissítéssel kapcsolatban
----------------------------------------------------------------

**Miért módosították a biztonsági közleményt 2010. június 22-én?**  
A módosított közlemény javított észlelési logikát tartalmaz ahhoz a problémához, ami miatt a 2010. június 8-án kiadott, a Microsoft .NET Framework 3.5 rendszerhez szóló biztonsági frissítést (KB982865) helytelenül telepítettnek minősítette a Microsoft Baseline Security Analyzer (MBSA) azokon a rendszereken, ahol a Microsoft .NET Framework 3.5 Service Pack 1 csomag már telepítve volt. Ettől függetlenül Microsoft .NET Framework 2.0 Service Pack 2 és a Microsoft .NET Framework 3.5 Service Pack 1 (KB979909) csomagok frissítései alkalmazásának szükségességét az MBSA helyesen állapítja meg. Ez csak észlelési módosítás, nem érinti az eredeti frissítésben foglalt fájlokat. A rendszerüket sikeresen frissített felhasználóknak nem kell újra telepíteniük a frissítést.

Az észlelési módosítás miatt a KB982865 frissítés újként tűnhet fel a Windows Server Update Services (WSUS) felhasználók számára. Ha nem alkalmaz automatikus jóváhagyási szabályokat, a frissítést manuálisan kell újból jóváhagynia. Ennek elmulasztása esetén a frissítést továbbra is felajánlják, még akkor is, ha az nem érvényes az adott rendszerekre. A változtatásról a [Microsoft Tudásbázis 894199. számú](http://support.microsoft.com/kb/894199)cikkében talál bővebb felvilágosítást.

**Mi az adatmanipulálási biztonsági rés?**  
Adatbiztonság szempontjából az adatmanipulálási biztonsági rés az adatok rosszindulatú módosítását teszi lehetővé. Ilyen lehet az állandó adatok, például egy adatbázis adatainak jogosulatlan módosítása vagy két számítógép között egy nyílt hálózaton, például az interneten zajló adatátvitel közben történő adatmódosítás. Ezen biztonsági rés esetében kifejezetten beavatkozásra utal, ha az adatok digitális aláírásához hatályos titkosítási aláírás használata esetén a digitális aláírás ellenőrzése sikertelen, mert az aláírás után ezeket az adatokat módosították. Szintén beavatkozásra utal, ha az aláírás ellenőrzése annak ellenére sikeres, hogy az adatokat manipulálták.

**Hogyan állapíthatom meg, hogy a Microsoft .NET Framework mely verziója van telepítve?**      
Egy rendszerre a .NET Framework több verzióját lehet telepíteni és futtatni, ráadásul a verziók bármilyen sorrendben telepíthetők. A .NET Framework aktuálisan telepített verziója több módon is meghatározható. További tudnivalókért tekintse át a [Microsoft Tudásbázis 318785. számú cikkét](http://support.microsoft.com/kb/318785) vagy az MSDN [A .NET Framework telepített verziójának meghatározása](http://msdn.microsoft.com/en-us/library/y549e41e.aspx) c. cikkét.

**Miért érinti a biztonsági rés a Microsoft .NET Framework 3.5 és Microsoft .NET Framework 3.5 Service Pack 1 rendszert egyes támogatott operációs rendszereken?**      
A Microsoft .NET Framework 3.5 a Microsoft .NET Framework 2.0 Service Pack 1 és a Microsoft .NET Framework 3.0 Service Pack 1 alösszetevőket is tartalmazza. A Microsoft .NET Framework 3.5 Service Pack 1 a Microsoft .NET Framework 2.0 Service Pack 2 és a Microsoft .NET Framework 3.0 Service Pack 2 alösszetevőket is tartalmazza. Egyes operációs rendszereken a Microsoft .NET Framework 3.5 és a Microsoft .NET Framework 3.5 Service Pack 1 telepítheti ezeket a támadható alösszetevőket.

**Az általam használt rendszerre a közleményben nem szereplő Microsoft .NET Framework verzió van telepítve. Érinti a biztonsági rés a konfigurációmat?**      
Teszteléssel állapították meg a közleményben tárgyalt szoftverkiadások érintettségét. A többi kiadás támogatási életciklusa végére ért vagy nem támogatott. A rendszerükön a Microsoft .NET Framework nem támogatott verzióját futtató ügyfelek számára a Microsoft .NET Framework telepített verziójának eltávolítása és újabb Microsoft .NET Framework verzióra való frissítés javasolt.

A Microsoft .NET Framework a Vezérlőpult **Programok telepítése és törlése** eszközével távolítható el. A Microsoft .NET Framework adott verzióinak eltávolításáról további információk a [Microsoft Tudásbázis 320122.](http://support.microsoft.com/kb/320112), a [Microsoft Tudásbázis 824643.](http://support.microsoft.com/kb/824643), illetve a [Microsoft Tudásbázis 908077.](http://support.microsoft.com/kb/908077) cikkében olvashatók.

**Tartalmaz a frissítés funkcionalitásbeli módosítást is?**      
Igen. A jelen közlemény **A biztonsági réssel kapcsolatos tudnivalók** c. részében felsorolt módosításokon felül a frissítés tartalmazza az ASP.NET kérések érvényesítése funkció alapvető biztonsági módosítását is, mellyel kiküszöbölhető, hogy a támadó megkerülje az ASP.NET kompatibilis weboldalakon alapértelmezett, alapvető biztonsági eljárást.

A problémát az okozza, hogy az ASP.NET kérések érvényesítése funkció nem ellenőriz megfelelően egy bizonyos karaktersorozatot. Az ASP.NET kérések ellenőrzése funkció nem helyettesítheti a nem megbízható bemeneti változókat korlátozó, hatályos érvényesítési réteget. Az internetes alkalmazások ASP.NET biztonsági funkcióiról a fejlesztők [Az ASP.NET beépített funkcióinak kihasználása az internetes támadások elhárításában](http://msdn2.microsoft.com/en-us/library/ms972969.aspx) c. MSDN-cikkben olvashatnak bővebben.

Ha egy weboldal nem rendelkezik hatályos érvényesítési réteggel a nem megbízható felhasználói beavatkozások korlátozására, a rést sikeresen kihasználó támadó tetszőleges, többek között ASP.NET tartalmakat is elhelyezhet az érintett weboldalon. Ez az alapvető biztonsági módosítás elhárítja ezt a közvetlenül jelzett hibát.

**Mi az ASP.NET kérések hitelesítése?**      
Az ASP.NET kérések ellenőrzése funkció alapvető biztonsági módosítása az ASP.NET alkalmazást futtató weboldalak általános bemeneti ellenőrzését látja el. Az ASP.NET kérések ellenőrzése funkció mindazonáltal nem helyettesítheti a nem megbízható bemeneti változókat korlátozó, hatályos érvényesítési réteget. Az internetes alkalmazások ASP.NET biztonsági funkcióiról a fejlesztők [Az ASP.NET beépített funkcióinak kihasználása az internetes támadások elhárításában](http://msdn2.microsoft.com/en-us/library/ms972969.aspx) és a [Nem megbízható az ASP.NET kérések hitelesítése](http://msdn.microsoft.com/en-us/library/ms998264.aspx) c. MSDN-cikkben olvashatnak bővebben.

**Honnan tudhatom, hogy az általam használt ASP.NET webes alkalmazásra vonatkozik-e ez az alapvető biztonsági módosítás?**      
A hiba nem érinti azokat az ASP.NET-tel fejlesztett webes alkalmazásokat, amelyek minden nem megbízható bemeneti változót a várt értékek vagy karakterek meghatározott tartományára korlátozzák. Az ASP.NET webes alkalmazások korlátozásával kapcsolatban lásd a [Microsoft tudásbázis 815155. cikkét](http://support.microsoft.com/kb/815155).

**Mi az ASP.NET?**      
Az ASP.NET a Microsoft .NET Framework rendszeren belül olyan technológiák gyűjteménye, amelyek segítségével a fejlesztők webes alkalmazásokat hozhatnak létre, illetve XML-alapú webes szolgáltatásokat valósíthatnak meg.

Szemben a hagyományos weboldalakkal, amelyek statikus HTML-kódot és parancsfájlokat kombinálnak, az ASP.NET lefordított, eseményvezérelt oldalakat alkalmaz. Mivel az ASP.NET egy webalapú alkalmazáskörnyezet, melynek futtatásához egy alapvető HTTP-funkcionalitással rendelkező webes kiszolgálóra van szükség, az ASP.NET az Internet Information Services (IIS) szolgáltatásban futtatható. További információt [a Microsoft hivatalos ASP.NET webhelyén](http://www.asp.net/) olvashat.

**Miért kell a problémát mélyreható biztonsági eljárással megszüntetni?**      
Az ASP.NET kérések ellenőrzésében azért kell alapvető biztonsági módosítást végrehajtani, mert ez a fejlesztői bemeneti ellenőrzésén felül extra biztonsági intézkedést nyújt. Csak a fejlesztő képes meghatározni, hogy egy adott alkalmazás esetében mi számít elfogadható bemenetnek. A mélységi biztonsági funkciókra akkor sem lehet támaszkodni, ha több ilyen réteg is rendelkezésre áll ahhoz, hogy jelentős mértékben megakadályozzák a kérdéses rendszer biztonsága ellen irányuló támadásokat. Ezért ha az ASP.NET kérések ellenőrzésében nem szerepel egy bizonyos karaktersorozat, az ellenőrzés maximális biztonsági hatása nem több, mintha a funkció teljes egészében le lenne tiltva.

**Mi a mélyreható biztonsági megoldás?**      
Az adatbiztonság területén a mélységben tagolt védelem olyan struktúrát jelent, melyben egy hálózat vagy egy rendszer biztonságát több védelmi réteggel védik a támadók ellen.

**Hol találhatók a fájlinformációs adatok?**      
A fájlinformációs adatokat lásd a **Biztonsági frissítés telepítése** rész hivatkozási táblázataiban.

Teszteléssel állapították meg a közleményben tárgyalt szoftverkiadások érintettségét. A többi kiadás támogatási életciklusa végére ért. Az XML-aláírás HMAC-levágása hitelesítését megkerülő biztonsági rés - CVE-2009-0217
-----------------------------------------------------------------------------------

Az adatmanipulálási biztonsági rés a Microsoft .NET Framework alkalmazásban van jelen és aláírt XML-tartalmak észrevétlen adatmanipulálását teheti lehetővé. Az egyedi alkalmazásokban a biztonsági hatás az adott felhasználási forgatókönyvtől függ. A biztonsági rés nem érinti azokat az eseteket, amikor aláírt XML-üzeneteket továbbítanak biztonságos csatornán (pl. SSL) keresztül.

A biztonsági rés standard bejegyzésének megtekintéséhez a normál biztonsági rés besorolási listáján lásd: [CVE-2009-0217](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0217).

#### Az XML-aláírás HMAC-levágása hitelesítését megkerülő biztonsági rés hibáit enyhítő tényezők - CVE-2009-0217

Az enyhítő tényezők közé sorolható egy beállítás, konfiguráció vagy általánosan bevált gyakorlat, amely alapértelmezett állapotában csökkentheti a biztonsági rés kihasználásának valószínűségét. A következő enyhítő tényezők hasznosnak bizonyulhatnak:

-   A Microsoft Office által végzett XML-aláírás ellenőrzéssel az adatmanipulálási biztonsági rést kihasználni kívánó támadó nem tud visszaélni.

#### Az XML-aláírás HMAC-levágása hitelesítését megkerülő biztonsági rés lehetséges megoldásai - CVE-2009-0217

A lehetséges megoldás olyan beállítást vagy konfigurációs módosítást takar, amely magát a biztonsági rést nem szünteti meg, azonban a frissítés telepítése előtt segít az ismert támadási lehetőségek kivédésében. A Microsoft alaposan megvizsgálta a következő lehetséges megoldásokat, és tájékoztatja a felhasználókat, amennyiben a megoldás csökkenti a működőképességet:

-   **Bizalmas adatok átviteléhez használjon biztonságos csatornát**

    Ha az alkalmazás bizalmas vagy az alkalmazás biztonságához kulcsfontosságú XML-adatok átvitelét végzi, az adatátvitelhez használjon biztonságos csatornát. Például Internet Information Server használata esetén a bizalmas vagy a biztonság szempontjából kulcsfontosságú alkalmazásokhoz vegye igénybe SSL használatát.

#### Gyakran feltett kérdések az XML-aláírás HMAC-levágása hitelesítését megkerülő biztonsági réssel kapcsolatban - CVE-2009-0217

**Mire terjed ki a biztonsági rés hatása?**      
A biztonsági rés az adatok manipulálásával a titkosítási aláírás megkerülését teszi lehetővé.

**Mi okozza a biztonsági rést?**      
A biztonsági rést a Microsoft .NET Framework alkalmazásban az XML-aláírás szintaxisára és feldolgozására irányuló W3C-javaslat (XMLDsig) okozza. Az XMLDsig szabványról további tudnivalók [Az XML-aláírás szintaxisa és feldolgozása (Második kiadás)](http://www.w3.org/tr/xmldsig-core/) c. W3C-javaslatban vagy a [Hibajegyzék az XML-aláíráshoz (Második kiadás)](http://www.w3.org/2008/06/xmldsigcore-errata.html) „E03 HMAC-levágás (CVE-2009-0217)” c. részében találhatók.

**Mi a Microsoft .NET Framework?**      
A Microsoft .NET Framework a Microsoft Windows operációs rendszer összetevője, mely szoftveralkalmazások és internetes szolgáltatások telepítését és futtatását teszi lehetővé. Tartalmazza az internetes szolgáltatásokhoz és alkalmazásokhoz (ASP.NET), az adatok hozzáféréséhez (ADO.NET), az intelligens ügyfélalkalmazásokhoz (Windows Forms) és még számos egyéb alkalmazáshoz szükséges technológiákat.

**Mi a HMAC?**      
A kivonatolás-alapú hitelesítő kód (Hash-based Message Authentication Code, HMAC) használatával megállapítható, hogy módosították-e a nem biztonságos csatornán keresztül küldött üzenetet, feltéve, ha a küldő és a címzett megosztott titkos kulccsal rendelkezik az üzenet sértetlenségének és hitelességének ellenőrzéséhez.

**Mi az XML?**      
Az XML (Extensible Markup Language) strukturált adatok leírásához formátumot biztosító, bővíthető leírókódnyelv. Az XML a World Wide Web Consortium (W3C) szabványa és az SGML (Standard Generalized Markup Language, szabványos általános jelölőnyelv) része. Az XML nyelvről további tudnivalók az MSDN adatfejlesztési központjának oldalán, az [XML kezdőlapján](http://msdn.microsoft.com/xml/) vagy az [Extensible Markup Language (XML) 1.0 (Ötödik kiadás)](http://www.w3.org/tr/rec-xml/) c. W3C-javaslatban találhatók.

**Mire használhatja a biztonsági rést a támadó?**      
A biztonsági rést sikeresen kihasználó támadó megkerülhet bizonyos titkosítási aláírásokat, melynek eredményeképp úgy manipulálhat aláírt XML-tartalmakat, hogy a címzett ne vegye észre a módosításokat. Az eredeti üzenet jelentésének teljes megváltoztatása is hamisítást jelent. HMAC-aláírással ellátott XML-tartalomra épülő alkalmazás írása esetén a biztonsági hatás az adott alkalmazás felhasználási forgatókönyvétől függ.

**Hogyan használhatja ki a támadó a biztonsági rést?**      
A támadó speciálisan kialakított XML-tartalmat küldhet egy érintett rendszerre. A Microsoft Office által végzett XML-aláírás ellenőrzéssel a támadó nem tud visszaélni.

**Mely rendszerekre jelenti a legnagyobb veszélyt a biztonsági rés?**      
Elsősorban a titkosítási műveletekhez kivonatolás-alapú hitelesítő kódot (Hash-based Message Authentication Code, HMAC) használó rendszerek és alkalmazások vannak kockázatnak kitéve. A Microsoft Office által végzett XML-aláírás ellenőrzéssel a biztonsági rést kihasználni kívánó támadó nem tud visszaélni.

**Mi a frissítés feladata?**      
A frissítés módosítja az XMLDsig ajánlás végrehajtását a Microsoft .NET Framework rendszerben.

**Ismert volt-e a nyilvánosság előtt ez a biztonsági rés a biztonsági közlemény kiadásának idején?**      
Igen. A biztonsági rés nyilvánosságra került, A biztonsági rés a [CVE-2009-0217](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0217) azonosítót kapta.

**A közlemény kibocsátásáig kapott-e a Microsoft a biztonsági rés kihasználásáról tájékoztató értesítést?**      
Nem. A Microsoft nem kapott olyan információt, hogy a biztonsági közlemény eredeti változatának megjelenéséig a biztonsági rést felhasználták volna nyilvános támadásra.

-   **Fájlverziószám ellenőrzése**

    Mivel a Microsoft Windows operációs rendszernek több verziója és kiadása is van, a lépések az egyes rendszereken eltérőek lehetnek. Ilyen esetben a megfelelő lépések végrehajtásához az adott termék dokumentációja ad segítséget.

    1.  Kattintson a **Start** menü **Keresés** parancsára.
    2.  A **Keresés eredménye** ablak **Keresés** ablaktáblájában válassza a **Minden mappa és fájl** elemet.
    3.  **A fájlnév része vagy egésze** mezőbe írja be a táblázatban szereplő megfelelő fájl nevét, majd kattintson a **Keresés** gombra.
    4.  A fájlok listájában kattintson jobb gombbal a megfelelő fájl nevére, majd kattintson a **Tulajdonságok** parancsra.  
        **Megjegyzés** Az operációs rendszer vagy a programok verziószámától függően elképzelhető, hogy a fájlinformációs táblázatban szereplő fájlok némelyike nem található meg a rendszerben.
    5.  A **Verzió** lapon hasonlítsa össze a rendszeren lévő fájl verziószámát a megfelelő táblázatban szereplő fájl verziószámával.  
        **Megjegyzés** A telepítés során a fájl verziószámán kívül más attribútumok is megváltozhatnak. Ezen attribútumok fájlinformációs táblázatban szereplő adatokkal való összevetése a frissítés alkalmazásának nem ajánlott ellenőrzési módja. Előfordulhat az is, hogy a telepítés során bizonyos esetekben a rendszer fájlokat nevez át. Ha a fájl- vagy verzióadatok nem találhatók, ellenőrizze a frissítés telepített példányát egyéb módszerek segítségével.

-   **Rendszerleíró kulcsok ellenőrzése**

    A biztonsági frissítés által telepített fájlokat a **Hivatkozási táblázatban** feltüntetett rendszerleíróadatbázis-kulcsok segítségével is ellenőrizheti.

    Nem biztos, hogy ezek a rendszerleíró kulcsok az összes telepített fájlt tartalmazzák. Ne feledje azt sem, hogy ha a rendszergazda vagy az OEM gyártó beépíti a biztonsági frissítést a Windows rendszer telepítési forrásfájljaiba, a program esetleg hibásan hozza létre a rendszerleíró kulcsokat.

**A frissítés telepítésének ellenőrzése**

-   **Microsoft Baseline Security Analyzer**

    A Microsoft Baseline Security Analyzer (MBSA) eszközzel ellenőrizheti, hogy a biztonsági frissítés telepítve van-e az érintett számítógépen. További tájékoztatásért tekintse meg a közlemény korábbi részének **Észlelési és telepítési eszközök, útmutatás** című fejezetét.

-   **Fájlverziószám ellenőrzése**

    Mivel a Microsoft Windows operációs rendszernek több verziója és kiadása is van, a lépések az egyes rendszereken eltérőek lehetnek. Ilyen esetben a megfelelő lépések végrehajtásához az adott termék dokumentációja ad segítséget.

    1.  Kattintson a **Start** menü **Keresés** parancsára.
    2.  A **Keresés eredménye** ablak **Keresés** ablaktáblájában válassza a **Minden mappa és fájl** elemet.
    3.  **A fájlnév része vagy egésze** mezőbe írja be a táblázatban szereplő megfelelő fájl nevét, majd kattintson a **Keresés** gombra.
    4.  A fájlok listájában kattintson jobb gombbal a megfelelő fájl nevére, majd kattintson a **Tulajdonságok** parancsra.  
        **Megjegyzés** Az operációs rendszer vagy a programok verziószámától függően elképzelhető, hogy a fájlinformációs táblázatban szereplő fájlok némelyike nem található meg a rendszerben.
    5.  A **Verzió** lapon hasonlítsa össze a rendszeren lévő fájl verziószámát a megfelelő táblázatban szereplő fájl verziószámával.  
        **Megjegyzés** A telepítés során a fájl verziószámán kívül más attribútumok is megváltozhatnak. Ezen attribútumok fájlinformációs táblázatban szereplő adatokkal való összevetése a frissítés alkalmazásának nem ajánlott ellenőrzési módja. Előfordulhat az is, hogy a telepítés során bizonyos esetekben a rendszer fájlokat nevez át. Ha a fájl- vagy verzióadatok nem találhatók, ellenőrizze a frissítés telepített példányát egyéb módszerek segítségével.

-   **Rendszerleíró kulcsok ellenőrzése**

    A biztonsági frissítés által telepített fájlokat a **Hivatkozási táblázatban** feltüntetett rendszerleíróadatbázis-kulcsok segítségével is ellenőrizheti.

    Nem biztos, hogy ezek a rendszerleíró kulcsok az összes telepített fájlt tartalmazzák. Ne feledje azt sem, hogy ha a rendszergazda vagy az OEM gyártó beépíti a biztonsági frissítést a Windows rendszer telepítési forrásfájljaiba, a program esetleg hibásan hozza létre a rendszerleíró kulcsokat.

**A frissítés telepítésének ellenőrzése**

-   **Microsoft Baseline Security Analyzer**

    A Microsoft Baseline Security Analyzer (MBSA) eszközzel ellenőrizheti, hogy a biztonsági frissítés telepítve van-e az érintett számítógépen. További tájékoztatásért tekintse meg a közlemény korábbi részének Észlelési és telepítési eszközök, útmutatás című fejezetét.

-   **Fájlverziószám ellenőrzése**

    Mivel a Microsoft Windows operációs rendszernek több verziója és kiadása is van, a lépések az egyes rendszereken eltérőek lehetnek. Ilyen esetben a megfelelő lépések végrehajtásához az adott termék dokumentációja ad segítséget.

    1.  Kattintson a **Start** menü **Keresés** parancsára.
    2.  A **Keresés eredménye** ablak **Keresés** ablaktáblájában válassza a **Minden mappa és fájl** elemet.
    3.  **A fájlnév része vagy egésze** mezőbe írja be a táblázatban szereplő megfelelő fájl nevét, majd kattintson a **Keresés** gombra.
    4.  A fájlok listájában kattintson jobb gombbal a megfelelő fájl nevére, majd kattintson a **Tulajdonságok** parancsra.  
        **Megjegyzés** Az operációs rendszer vagy a programok verziószámától függően elképzelhető, hogy a fájlinformációs táblázatban szereplő fájlok némelyike nem található meg a rendszerben.
    5.  A **Verzió** lapon hasonlítsa össze a rendszeren lévő fájl verziószámát a megfelelő táblázatban szereplő fájl verziószámával.  
        **Megjegyzés** A telepítés során a fájl verziószámán kívül más attribútumok is megváltozhatnak. Ezen attribútumok fájlinformációs táblázatban szereplő adatokkal való összevetése a frissítés alkalmazásának nem ajánlott ellenőrzési módja. Előfordulhat az is, hogy a telepítés során bizonyos esetekben a rendszer fájlokat nevez át. Ha a fájl- vagy verzióadatok nem találhatók, ellenőrizze a frissítés telepített példányát egyéb módszerek segítségével.

-   **Rendszerleíró kulcsok ellenőrzése**

    A biztonsági frissítés által telepített fájlokat a **Hivatkozási táblázatban** feltüntetett rendszerleíróadatbázis-kulcsok segítségével is ellenőrizheti.

    Nem biztos, hogy ezek a rendszerleíró kulcsok az összes telepített fájlt tartalmazzák. Ne feledje azt sem, hogy ha a rendszergazda vagy az OEM gyártó beépíti a biztonsági frissítést a Windows rendszer telepítési forrásfájljaiba, a program esetleg hibásan hozza létre a rendszerleíró kulcsokat.

    2.  Amikor a fájl megjelenik a **Programok** között, jobb egérgombbal kattintson a fájlnévre, majd a **Tulajdonságok** lehetőségre.
    3.  Az **Általános** lapon hasonlítsa össze a fájlméretet a közlemény Tudásbázis cikkében ismertetett fájlinformációs táblázatokkal.
    4.  Az adatok összevetéséhez (pl. fájlverzió, a módosítás dátuma) a **Részletek** lapra is kattinthat, a közlemény Tudásbázis cikkében ismertetett fájlinformációs táblázatok alapján.
    5.  Kattinthat a **korábbi verziókat** felsoroló lapra is, majd összehasonlíthatja a korábbi fájlverzió adatait az aktuális, frissített fájléval.
